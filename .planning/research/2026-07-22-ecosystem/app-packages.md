# Ecosystem research — open-source packages for the polytoken app

Date: 2026-07-22. Scope: packages benefiting the **application itself** (not meta/dev tooling).
Grounded in repo reality first (paths cited), then external research (URLs cited). Assumptions are labeled.

Repo facts this report is anchored on:

- Canvas: `@xyflow/react ^12.11.0` + `@dagrejs/dagre` + `zustand` (`apps/web/package.json`); canvas surface at `apps/web/src/app/chat/_canvas`, layouts persisted in `packages/db/src/schema/chat-canvas-layouts.ts`.
- Circular treemap is **greenfield** — zero circle-pack/treemap code anywhere (`.planning/research/2026-07-22-META-AUDIT.md` line 72).
- The "in-house" spreadsheet (`packages/ui/src/spreadsheet-grid/`) is **not from-scratch** — it is a wrapper around **AG Grid Community `^35.1.0`** (`packages/ui/package.json`; `SpreadsheetGrid.tsx` imports `ag-grid-community`/`ag-grid-react`), with hand-rolled range selection, clipboard, find, and conditional formatting hooks. It is currently imported by no surface (META-AUDIT line 73).
- Entity resolution: BlendedRAG — pgvector halfvec dense arm + pg_trgm lexical arm fused with RRF(k=60), suggest-only (`apps/email-listener/app/infrastructure/supabase/entity_resolution_repository.py`, `app/application/use_cases/resolve_entity_candidates.py`, migrations `0017`, `0039`).
- Email parsing: pure-stdlib `email` with `policy.default`, `parseaddr`/`getaddresses` (`apps/email-listener/app/domain/services/mime_parser.py`); raw RFC 5322 bytes come from S3 (`app/infrastructure/s3/raw_email_store.py`), SES inbound bucket has a lifecycle config but versioning is not declared (`infrastructure/aws/ses.tf`). `requires-python = ">=3.11"` (`apps/email-listener/pyproject.toml`).
- Drive/vault: `apps/web/src/app/files` uploads to Supabase Storage via signed URLs (`_lib/use-vault-upload.ts`). No file-version table exists in `packages/db/src/schema/` (checked: no vault/version schema).
- Web stack: Next `^15.3.3`, React `^19.2.7`, tRPC `11.8.0`, TanStack Query `^5.62`, superjson, zustand.

## 1. Circle packing / zoomable pack for the circular-treemap canvas nodes

**Recommendation: `d3-hierarchy` (`d3.pack()`) as the layout engine only, rendered through xyflow's own node/viewport system. Do NOT adopt a charting library's zoom (d3-zoom, nivo's built-in zoom) — it fights xyflow's pan/zoom.**

Options assessed:

| Option | License | Fit |
|---|---|---|
| `d3-hierarchy` pack | ISC | **Best.** Pure layout math (returns `x`, `y`, `r` per node, no DOM). Feeds xyflow nodes directly. Tiny, zero-dep, the de-facto standard. [d3js.org/d3-hierarchy/pack](https://d3js.org/d3-hierarchy/pack) |
| `@nivo/circle-packing` | MIT | Good reference impl (SVG/HTML/canvas renderers, zoom, `circleComponent` override), but it owns its own zoom + rendering — wrong shape for embedding inside an xyflow canvas node. Useful for a non-canvas "traditional UI" rendering of the same data. [nivo.rocks/circle-packing](https://nivo.rocks/circle-packing/), [npm](https://www.npmjs.com/package/@nivo/circle-packing) |
| `@visx/hierarchy` | MIT | Thin React wrapper over d3-hierarchy `Pack`. Fine, but adds a dependency for what is ~30 lines over raw d3-hierarchy. [codesandbox example](https://codesandbox.io/s/circle-packing-fiddle-vwhd6) |
| Observable zoomable-circle-packing | ISC (example code) | The canonical zoom-interaction reference (click-to-zoom via interpolated transform). Port the *interaction idea*, not the d3-zoom code. [observablehq.com/@d3/zoomable-circle-packing](https://observablehq.com/@d3/zoomable-circle-packing) |

**xyflow-compatible architecture (two viable patterns):**

1. **One custom node = one circreemap** (recommended first). A single xyflow custom node whose body is an SVG rendered from a `d3.pack()` layout. Zoom-into-circle is an internal state change (re-root the hierarchy and animate `viewBox`, à la Observable's interpolator) — xyflow's viewport zoom stays untouched. This matches the vision doc's "circreemaps can have components inside it that the agent uigenerates" (`.planning/prompts/2026-07-22-vision-and-handoff.md` line 185): each packed circle can portal an agent-generated genui component once its on-screen radius crosses a threshold (semantic zoom).
2. **Circles as first-class xyflow child nodes** using `parentId` + `extent: 'parent'`, positions computed by `d3.pack()` exactly the way dagre positions are computed today (xyflow explicitly documents d3-hierarchy as a layout engine option: [reactflow.dev/examples/layout/auto-layout](https://reactflow.dev/examples/layout/auto-layout), [dagre example](https://reactflow.dev/examples/layout/dagre)). Heavier (thousands of nodes = thousands of React nodes), but circles become selectable/draggable/connectable citizens of the canvas. Adopt only after pattern 1 proves the UX; consider it for the top 2–3 levels only, with pattern-1 SVG for deep leaves.

Performance note: for drive treemaps over ~2–5k visible circles, render leaves to `<canvas>` inside the custom node and keep only interactive circles as DOM/SVG (this is exactly why `@nivo/circle-packing` ships a canvas renderer). d3-force-based packing ([react-graph-gallery](https://www.react-graph-gallery.com/circular-packing)) is for flat one-level packing — not needed for hierarchies.

**Adopt:** `d3-hierarchy` (and nothing else) as a new `apps/web` dependency. Assumption: bundle cost (~10 kB min) is acceptable; it is.

## 2. Spreadsheet stack — build on in-house vs adopt

**Recommendation: keep building on the in-house AG Grid Community wrapper. Do not adopt handsontable, jspreadsheet, revogrid, or univer now. Re-evaluate univer only if/when full formula-engine + realtime co-editing become requirements.**

First, correct the framing: `packages/ui/src/spreadsheet-grid` is ~2k lines of product glue *on top of AG Grid Community (MIT)* — column dialogs, conditional-formatting rules, tenant-persisted column state, paste-preview, find bar. The hand-rolled hooks (`use-range-selection`, `use-grid-clipboard`, `use-grid-find`) exist precisely because cell range selection and Excel-like clipboard are **AG Grid Enterprise-only** features ([Community vs Enterprise](https://www.ag-grid.com/react-data-grid/community-vs-enterprise/), [cell selection docs](https://www.ag-grid.com/javascript-data-grid/cell-selection/), [pricing](https://www.ag-grid.com/license-pricing/)). That work is done and tested; switching stacks throws it away.

| Stack | License | Verdict for polytoken |
|---|---|---|
| **In-house (AG Grid Community wrapper)** | MIT (Community) | **Keep.** Already themed to the design system (`themeQuartz.withParams` + tokens), already has editors/CF/clipboard/find. The real gap is not the grid — it's persistence schema + agent wiring (META-AUDIT: "100% unwired"). |
| Handsontable | **Proprietary/commercial.** Free only for evaluation/non-commercial; production commercial use requires a paid license ([handsontable.com/docs/.../software-license](https://handsontable.com/docs/react-data-grid/software-license/), [github](https://github.com/handsontable/handsontable)) | **Reject.** Licensing alone disqualifies it for a product that may commercialize. |
| Glide Data Grid | MIT ([overview](https://best-of-web.builder.io/library/glideapps/glide-data-grid)) | Strongest *alternative* if AG Grid Community hits a wall: canvas-rendered, handles millions of rows, built for Glide's no-code product. But it is a *grid*, not a spreadsheet — you'd re-write all the glue (editors, CF, menus) against a different API for a perf level you don't yet need. Hold as plan B for >100k-row datasets. |
| RevoGrid | MIT, open-core (some plugins paid) ([rv-grid.com/guide/licensing](https://rv-grid.com/guide/licensing), [github ~3.1k stars](https://github.com/revolist/revogrid)) | Stencil-based web component with React wrapper; smaller community than AG Grid; no advantage over what exists. Skip. |
| Univer | Apache-2.0 core, optional commercial pro ([univer blog comparison](https://blog.univer.ai/posts/univer-vs-handsontable-a-complete-comparison-of-next-generation-online-spreadsheet-engines/), [10-best roundup](https://blog.univer.ai/posts/10-best-spreadsheet-components-for-developers-in-2025/)) | The only candidate that adds a genuinely new capability class: real formula engine, sheets/docs model, OT-based realtime collaboration. Heavy (its own document model + renderer, large bundle) and it wants to own the whole surface — poor fit for embedding inside canvas panels today. **Re-evaluate at the teams/sharing milestone** if "Excel-like" comes to mean formulas + co-editing rather than structured data tables. |
| Jspreadsheet CE | CE v5 is MIT; Pro is paid ([npm](https://www.npmjs.com/package/jspreadsheet-ce), [LICENSE](https://github.com/jspreadsheet/ce/blob/master/LICENSE)) | Lightweight jQuery-era architecture, upsell-driven feature split. No advantage. Skip. |

Where to spend instead (per vision doc line 167 — agent-suggested tables): a `data_sources`/`records` persistence schema in `packages/db`, tRPC routers, and a capability in `packages/capabilities` for agent "suggest table / suggest row-update" — then mount the existing grid. Assumption: expected dataset sizes are ≤~50k rows per table near-term; AG Grid Community's DOM virtualization is fine there.

## 3. Entity resolution / dedupe libraries vs BlendedRAG

**Recommendation: keep BlendedRAG (pgvector + pg_trgm + RRF) as the online, suggest-only candidate generator. Adopt `splink` (MIT) later as an offline batch scorer for backfill/audit if precision becomes a measured problem. Reject zingg (AGPL + Spark) and dedupe (label-hungry) for this codebase.**

| Option | License | Fit vs current |
|---|---|---|
| **Current BlendedRAG** | in-house | Right shape for the product: online, per-instance, top-N suggestions with human confirm (`resolve_entity_candidates.py` is deliberately read-only, D-05). Embeddings capture semantic alias equivalence ("Acme Corp" ≈ "Acme Inc.") that classic ER blocking misses. Its weakness: no calibrated match *probability* and no learned per-field weights — RRF rank fusion is unitless. |
| [splink](https://github.com/moj-analytical-services/splink) (UK Ministry of Justice) | MIT | **Best complement.** Fellegi–Sunter probabilistic linkage with unsupervised EM training, DuckDB backend (no Spark needed), dedupes 7M records in minutes ([Robin Linacre](https://www.robinlinacre.com/fast_deduplication/), [Medium](https://medium.com/data-science-collective/deduplicating-7-million-records-in-two-minutes-with-splink-4b1a87035a85)), interactive diagnostics ([docs](https://moj-analytical-services.github.io/splink/index.html)). Use as a *batch job* over `entity_instances` (names, email addresses, domains) to (a) generate calibrated merge-probability scores that re-rank BlendedRAG candidates, and (b) audit for missed merges. Runs fine inside the FastAPI worker's uv env; Clean Architecture home: new infrastructure adapter behind the existing `EntityResolutionRepository` port. |
| [dedupe](https://github.com/dedupeio/dedupe) | MIT | Active-learning: requires a human labeling session per model. Polytoken already has a human-confirm loop feeding `entity_type_corrections`/promote-on-confirm — those confirmations are better spent as splink priors than as dedupe training UI. Skip. |
| [zingg](https://github.com/zinggAI/zingg) | **AGPL-3.0**, Spark-based | AGPL is a real constraint for a possibly-commercial SaaS, and Spark is absent from the stack. Reject. ([Tilores comparison](https://tilores.io/content/best-open-source-entity-resolution-and-record-linkage-libraries-splink-zingg-dedupe-and-when-to-move-beyond-them/)) |

Cheap wins before adding any library: log per-arm ranks (vector vs trgm) into the resolution telemetry so RRF quality is measurable; add an email-domain blocking arm (exact/da domain match) as a third RRF list — near-zero code given `_merge_rrf` already takes N lists (`entity_resolution_repository.py`).

## 4. File versioning / backup for the drive product

Current reality: vault files go to Supabase Storage via signed upload URLs; **Supabase Storage does not support S3 bucket versioning — deletes are permanent** ([S3 compatibility matrix](https://supabase.com/docs/guides/storage/s3/compatibility), [storage repo](https://github.com/supabase/storage)). The vision doc demands "robust systems against catastrophic irrecoverable data loss" (line 181) ahead of a 500 GB OneDrive migration (line 177). So versioning must be **application-level**.

**Recommended pattern: content-addressed blobs + a version table, plus an out-of-band replica.**

1. **Content-addressed storage (CAS) in the app.** Store blobs keyed by `sha256(content)` (e.g. `blobs/{hash[0:2]}/{hash}`); a `files` table maps logical path → ordered list of `(version_id, blob_hash, size, mtime, author)`. This is the git/restic model: "save a new version" = insert a row; identical content across versions/users deduplicates for free; delete = tombstone row, never a blob delete (GC only via a mark-and-sweep job with a retention window). This is the single highest-leverage schema decision and costs nothing extra on Supabase Storage. (Same pattern Supabase community guidance suggests: path-versioned objects + DB-managed relationships — [bootstrapped.app guide](https://bootstrapped.app/guide/how-to-implement-file-versioning-in-supabase-storage).)
2. **Chunking — defer.** restic/borg-style content-defined chunking (CDC) dedupes *within* large mutating files. For a drive of mostly write-once documents/images, whole-file CAS captures ~all the win with far less complexity. Adopt CDC only if large frequently-edited files (video projects, DB dumps) show up in real usage. If needed, the reference designs are [restic's design doc](https://restic.readthedocs.io/en/stable/100_references.html) (CDC + encrypted pack files + snapshot tree) — steal the architecture, not the binary.
3. **Off-provider replica (the actual catastrophe guard).** App-level versioning doesn't survive a Supabase project loss or a bad RLS bug. Nightly `rclone sync` (MIT, [rclone.org](https://rclone.org/)) of the storage bucket (Supabase Storage speaks the S3 protocol — [blog](https://supabase.com/blog/s3-compatible-storage)) to a **true S3 bucket with `aws_s3_bucket_versioning` + Object Lock (compliance mode) + lifecycle to Glacier** — the polytoken Terraform already manages S3 (`infrastructure/aws/ses.tf` has lifecycle config; add versioning there for the SES bucket too, it is currently unversioned). S3 versioning + Object Lock is the industry-standard immutable/ransomware-resistant tier ([AWS versioning docs](https://docs.aws.amazon.com/AmazonS3/latest/userguide/Versioning.html)).
4. Postgres side: PITR/WAL backups already come with Supabase paid tiers; the version *metadata* must be in the same backup domain as the blobs' hashes so restore is coherent. Assumption: current Supabase plan includes PITR — verify before the migration.

Alternative considered and rejected for now: running restic/rustic itself against user drives (it's a backup *tool*, not an embeddable library) and SeaweedFS/MinIO self-hosting (operational burden misaligned with a solo founder).

## 5. Email parsing hardening

Current parser (`mime_parser.py`) is clean stdlib, but three concrete hardening items:

1. **CVE-2023-27043 — directly applicable.** `email.utils.parseaddr`/`getaddresses` (both used in `mime_parser.py` lines 51–53, 102) misparse addresses containing special characters, letting a crafted `From:` display a trusted domain while the real addr-spec differs — this feeds `sender_address`, which feeds sender profiles and entity resolution, i.e. an attacker can pollute the knowledge graph identity layer. Patched Pythons add a `strict` kwarg (strict behavior default-on in fixed releases) ([CPython issue #102988](https://github.com/python/cpython/issues/102988), [Red Hat advisory](https://access.redhat.com/articles/7051467), [cvedetails](https://www.cvedetails.com/cve/CVE-2023-27043/)). Action: pin runtime Python to a patched 3.11.x (repo only requires `>=3.11`), pass `strict=True` explicitly, and treat `(name, '')` results as parse failures rather than empty senders.
2. **Defect surfacing.** stdlib parsing silently tolerates malformed MIME. [`mail-parser` (SpamScope)](https://github.com/SpamScope/mail-parser) (Apache-2.0, built on the same stdlib) exposes a `defects` list (RFC violations, malformed boundaries) designed for security analysis — either adopt it as the infrastructure-layer parser feeding the existing `ParsedEmail` domain object, or copy the cheap version: read `msg.defects`/part defects after `message_from_bytes` and log + persist them (fits the "email-analysis hardening / surface swallowed errors" priority in META-AUDIT line 95).
3. **Authenticity + reply hygiene.** (a) Verify or at least record SES's `Authentication-Results` (SPF/DKIM/DMARC verdicts) per message before trusting sender identity for entity promotion; for independent verification, [dkimpy](https://pypi.org/project/dkimpy/) (BSD-ish/zlib) does DKIM+ARC. (b) For quoted-reply stripping before LLM analysis/embedding (reduces token waste and stops quoted-text prompt injection replay), Mailgun's [talon](https://github.com/mailgun/talon) (Apache-2.0) is the reference algorithm but is stale — assumption flagged: vet maintenance before adopting; the regex-based heuristics are small enough to vendor. HTML body sanitization is already covered web-side (dompurify + rehype-sanitize in `apps/web`).

## 6. Next.js 15 / React 19 snappiness patterns

Targets vision doc line 189 ("reducing frontend clunkiness... snappier"). Ordered by leverage-per-risk on the current stack (Next 15.3, React 19.2):

1. **Optimistic mutations everywhere user-visible.** tRPC 11 + TanStack Query 5 canonical pattern: `onMutate` → `utils.x.cancel()` → snapshot → `utils.x.setData()` → rollback in `onError` ([TanStack optimistic updates guide](https://tanstack.com/query/latest/docs/framework/react/reference/useMutation), [tRPC discussions](https://discord-questions.trpc.io/m/1202009982120501258)). The repo already has mutation hooks (`use-role-mutations.ts`, `use-region-edit.ts`, etc. in `apps/web/src/app/emails/[id]/_components/`) — audit which ones lack `onMutate` optimism; that's where "clunky" lives. React 19's `useOptimistic` is the right tool only for component-local action state (e.g. chat send), not for the shared tRPC cache ([dev.to overview](https://dev.to/arafatruetbd/react-19-optimistic-ui-with-useoptimistic-11mn)).
2. **Router cache `staleTimes` + deliberate prefetching.** Next 15 defaults `staleTimes.dynamic` to 0 — every navigation refetches. Set `experimental.staleTimes = { dynamic: 30–60 }` for the app's dashboard-like surfaces and use `router.prefetch()` / hover-prefetch for inbox→detail transitions ([staleTimes docs](https://nextjs.org/docs/app/api-reference/config/next-config-js/staleTimes), [prefetching guide](https://nextjs.org/docs/app/guides/prefetching)). Pair with TanStack Query `prefetchQuery` on hover for tRPC data ([TanStack prefetching](https://tanstack.com/query/latest/docs/framework/react/guides/prefetching)).
3. **React Compiler.** Stable 1.0 since Oct 2025 ([react.dev announcement](https://react.dev/blog/2025/10/07/react-compiler-1)); experimental flag in Next 15, stable integration in Next 16 ([Next 16 blog](https://nextjs.org/blog/next-16)). The canvas + grid code is memoization-heavy hand-written `useMemo`/`useCallback` (see `SpreadsheetGrid.tsx`) — the compiler removes whole classes of missed-memo re-renders. Enable behind `experimental.reactCompiler` in a verify build (`build:local`, per CLAUDE.md) and measure with the existing geometry gates. Assumption: upgrade to Next 16 is on the roadmap; compiler is the strongest single argument to schedule it.
4. **View Transitions — experimental, adopt narrowly.** React `<ViewTransition>` + Next `experimental.viewTransition: true` gives compositor-thread page/element transitions ([Next viewTransition config](https://nextjs.org/docs/app/api-reference/config/next-config-js/viewTransition), [view transitions guide](https://nextjs.org/docs/app/guides/view-transitions)); still marked experimental/not-for-production in Next 15. Use for low-risk shared-element polish (inbox row → email detail header) behind the flag; do not build core UX on it yet. Note interaction risk with the `motion` library already in `packages/ui` — pick one owner per transition.
5. **Streaming + Suspense boundaries** on data-heavy routes (`/emails/[id]`, `/knowledge`): move slow tRPC server calls behind `<Suspense>` so shell paints immediately (standard Next 15 streaming; [Next 15 blog](https://nextjs.org/blog/next-15)). Also switch tRPC links to `httpBatchStreamLink` if not already, so batched queries stream in as they resolve.
6. **Canvas-specific:** keep xyflow node components referentially stable (module-scope `nodeTypes`), and use zustand selectors (already in stack) rather than context broadcasting for per-node state — the #1 xyflow jank source is nodeTypes identity churn (assumption from xyflow docs guidance; verify in `_canvas` code during implementation).
