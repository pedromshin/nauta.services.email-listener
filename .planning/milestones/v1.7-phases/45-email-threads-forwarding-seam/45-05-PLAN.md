<objective>
Build the FastAPI-side forwarding seam (THRD-04): a `ForwardingAddressResolver` port + Supabase adapter that maps a recipient `u-{token}@<domain>` address to its owning `user_id`, the SES envelope recipients threaded from the SNS webhook into ingest, and the resolved user used to anchor a newly-created importer. This makes ingest correct for the multi-user future (importers.user_id is NOT NULL since Phase 44) and guarantees Gmail's forwarding-verification email lands in the user's pipeline unquarantined.

Purpose: The forwarding address is the personal ingestion seam. Resolving its secret token to a user is what lets a new sender-domain importer be created under the right owner, and what keeps the Gmail verification handshake working (the runbook, Plan 45-06, depends on that mail being ingestible).
Output: Token resolver port + adapter, recipient threading through ingest, user-anchored importer creation, and tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SES recipient/envelope → token resolver | recipient addresses are attacker-influenceable; a bad token must not disclose or invent a user |
| resolved user_id → importer creation | the anchor determines tenancy of ingested mail; must come from a real forwarding_addresses row |
| inbound forwarding-verification mail → pipeline | must be ingested (readable), not silently dropped/quarantined |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-45-05-01 | Spoofing | guessing/enumerating another user's token to steer mail into their tenant | mitigate | token is a high-entropy secret (generated CSPRNG-side, Plan 45-06); resolve returns None for unknown tokens (fail-closed); an attacker sending to a known token only injects mail into that user's own inbox — inherent to email forwarding, mitigated by token secrecy |
| T-45-05-02 | Elevation of Privilege | token resolves to user A but mail anchored to user B | mitigate | user_id flows straight from the forwarding_addresses row to importer creation; no client input involved; importers.user_id is NOT NULL FK to auth.users |
| T-45-05-03 | Denial of Service | resolver exception aborts ingestion / retry storm | mitigate | forwarding resolution is best-effort (guarded try/except → None); SNS handler already returns 200 unconditionally |
| T-45-05-04 | Repudiation/Tampering | Gmail verification code silently dropped, breaking onboarding | mitigate | verification mail is ingested through the normal path under the resolved user; test asserts email_repo.save is called for the google-sender/token-recipient case |
| T-45-05-SC | Tampering | dependency install | accept | No new packages; uses stdlib + existing supabase Client |
</threat_model>
