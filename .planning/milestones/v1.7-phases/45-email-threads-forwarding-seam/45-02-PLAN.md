<objective>
Build the pure thread-grouping domain service that turns a set of parsed emails into thread groups, resilient to Gmail-UI forwards. This is the algorithmic heart of THRD-01 (Union-Find over RFC headers) and THRD-02 (forwarded-mail fallback tiers). It has zero DB/I/O dependencies so it is exhaustively unit-testable and reused by both the incremental ingest resolver (Plan 45-03) and the backfill (Plan 45-03).

Purpose: A wrongly-merged thread is user-visible corruption; a fragmented thread is recoverable. The algorithm must be conservative and provably so via tests. Isolating it as a pure function lets us pin behavior with fixtures before wiring persistence.
Output: `thread_grouping.py` (Union-Find + Tier 1 + Tier 2) and a test suite driven by real/representative `.eml` fixtures.
</objective>

<feature>
  <name>Thread grouping domain service</name>
  <files>apps/email-listener/app/domain/services/thread_grouping.py, apps/email-listener/tests/domain/services/test_thread_grouping.py</files>
  <behavior>
    Tier 0 (headers, THRD-01): emails A,B,C where B.in_reply_to == A.message_id and C.references_ids contains A.message_id → one group {A,B,C}. Two disjoint conversations → two groups. Union-Find so a mid-arriving email that bridges two partial clusters merges them into one.
    Tier 1 (embedded ids, THRD-02): a forwarded email whose References/In-Reply-To are stripped but whose body contains the original "Message-ID: <...>" of an existing thread member → joins that thread.
    Tier 2 (conservative fallback, THRD-02): a forwarded email with NO usable header link and NO embedded id, but whose normalized subject equals an existing member's normalized subject AND received within the bounded window → joins. If the subject is generic/empty, or outside the window, or ambiguous (matches two different threads) → DO NOT merge; start a new group (false-split beats false-merge).
    normalize_subject: strips repeated leading Re:/Fwd:/Fw:/Enc:/Res:/Fw: tokens (case-insensitive, whitespace-tolerant), collapses whitespace, lowercases; empty/whitespace subject → "".
    extract_embedded_message_ids: scans body_text and body_html for `Message-ID:` header lines (angle-bracketed ids) as embedded in Gmail forward blocks; returns a de-duplicated tuple; returns () when none.
  </behavior>
  <implementation>
    Define `@dataclass(frozen=True) class ThreadableEmail` with fields id:str, message_id:str|None, in_reply_to:str|None, references_ids:tuple[str,...], subject:str|None, received_at:datetime, body_text:str|None, body_html:str|None.
    Implement a small internal Union-Find (parent dict keyed by email id, find with path compression, union). Build message_id → email-id index. For each email: union with any email whose message_id is in (in_reply_to ∪ references_ids ∪ extract_embedded_message_ids(body)), and with any email that references THIS email's message_id (bidirectional). This is Tier 0 + Tier 1.
    Tier 2 pass (only for emails still alone in their component): group candidate by (importer implicit — caller passes a single importer's emails) normalize_subject match within `window: timedelta` of an existing group's latest received_at; skip when normalized subject is "" or when the match is ambiguous across ≥2 distinct components.
    `group_emails(emails: Sequence[ThreadableEmail], *, window: timedelta) -> list[tuple[str, ...]]` returns groups of email ids; group order and intra-group order deterministic (sort by received_at then id) so callers (backfill) are idempotent. window default choice is Claude's discretion (CONTEXT) — pick a conservative bounded value (e.g. 14 days) and name it as a module constant with a comment.
    Do NOT hand-roll an external threading lib (jwzthreading abandoned 2010 — CONTEXT). Pure stdlib only.
  </implementation>
</feature>

|-----------|----------|-----------|-------------|-----------------|
| T-45-02-01 | Spoofing | forged In-Reply-To/References/embedded Message-ID to force a wrong merge | mitigate | grouping operates only within a single importer's email set (caller-scoped, Plan 45-03); cross-importer/cross-tenant merge is structurally impossible; Tier 2 refuses ambiguous/empty-subject matches (false-split beats false-merge) |
| T-45-02-02 | Denial of Service | pathological references_ids / huge forwarded body | mitigate | Union-Find is near-linear; extract_embedded_message_ids uses a bounded regex over decoded body only; no recursion on attacker input |
| T-45-02-SC | Tampering | dependency install | accept | Pure stdlib only — no external threading library (jwzthreading rejected); no package installs in this plan |
