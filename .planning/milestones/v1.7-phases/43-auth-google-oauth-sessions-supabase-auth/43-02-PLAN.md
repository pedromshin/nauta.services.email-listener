<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Next middleware | Every request (incl. attacker-crafted paths/params) crosses here before reaching a protected surface |
| OAuth provider → /auth/callback | The `code` and `next` params return from Google/Supabase and are attacker-influenceable |
| session cookie → authorization | Cookie must be server-verified, not trusted from a raw parse |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-43-P2-01 | Tampering (open redirect) | `redirectTo` / callback `next` param | mitigate | `safeNextPath` rejects `//` and absolute URLs → fallback `/`; used in both middleware redirect and callback; unit-tested |
| T-43-P2-02 | Spoofing / Repudiation (session fixation, CSRF) | `/auth/callback` | mitigate | PKCE flow (code_verifier in httpOnly cookie) + `exchangeCodeForSession` rotates the session; server-verified |
| T-43-P2-03 | Spoofing (unverified session) | middleware, callback, signout | mitigate | All server contexts use `getUser()` (via updateSession/server client), never `getSession()` alone |
| T-43-P2-04 | Elevation (auth bypass via matcher gap) | `middleware.ts config.matcher` | mitigate | Matcher guards all app surfaces; only `/login`, `/auth`, `_next`, static, `/api` excluded; `resolveAuthRedirect` unit test enumerates protected vs public paths |
| T-43-P2-05 | Repudiation (stale session after sign-out) | `/auth/signout` | mitigate | Server-side `signOut()` clears httpOnly cookies; redirect to `/login` |
| T-43-P2-06 | Information Disclosure | callback error branch | mitigate | Failure redirects to `/login?error=auth` without echoing upstream/provider error detail |
</threat_model>
