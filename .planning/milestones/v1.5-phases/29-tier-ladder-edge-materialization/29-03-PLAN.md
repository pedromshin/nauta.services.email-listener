<objective>
Implement the KnowledgeSynthesizerService: on a confirmation it materializes one knowledge_node per
confirmed region and a supersede-safe set of EXTRACTED-tier edges — the confirmed-component provenance
anchor, co-occurrence edges to the email's other confirmed entity components, and (when resolvable) an
"about" edge to the region's entity instance.

Purpose: SYNTH-03 requires re-confirm/supersede to update a region's edges with no duplicate or orphaned
rows, consistent with the Phase-6 supersede-versioning convention. The decisive design choice: the node
is 1:1 with the confirmed region (scope_ref_id = component_id), so `deactivate_edges_for_node(node_id)`
(the port frozen in 29-02) deactivates exactly and only this region's edges — never another region's —
making deactivate-then-insert a clean, orphan-free supersede.
Output: two entity-instance read methods, the synthesizer service, and AsyncMock-port unit tests.
</objective>

| OCR-derived text/tokens → node content + provenance | untrusted document tokens enter node content and edge provenance |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-07 | Elevation of privilege | cross-tenant edge write | mitigate | Node is upserted with importer_id derived from the loaded component (never client-supplied); every edge's source_node_id is that node — an edge can only ever hang off a node in the confirmer's tenant |
| T-29-08 | Tampering | supersede correctness | mitigate | Node is 1:1 with the region (scope_ref_id=component_id) so deactivate_edges_for_node touches only this region's edges; deactivate uses is_active=false, never DELETE — asserted in tests |
| T-29-09 | Denial of service | co-occurrence fan-out | mitigate | Co-occurrence source is email-scoped (find_confirmed_entity_components_for_email), bounding edge count to entities in one email, not the whole importer |
| T-29-SC | Tampering | package installs | accept | No new npm/pip/cargo packages introduced |
</threat_model>
