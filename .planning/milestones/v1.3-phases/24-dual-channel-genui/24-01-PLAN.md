<objective>
Build the persistence + safety-primitive spine for agent↔user widget round-trips: a
`chat_widget_interactions` table that stores each pending widget's declared response schema and
tracks its lifecycle (`pending → submitted | superseded | stale`) with a DB-level double-submit
lock, plus the pure re-validation function and the two new typed message-part shapes that every
later plan reads and writes.

Purpose: DCUI-03's three guarantees (server-side re-validation against the STORED schema, DB-level
double-submit lock, turn-bound staleness) and DCUI-04's persistence all rest on this foundation. No
UI, no tool, no endpoint here — just the storage, the lock, the validator, and the part-shape
contract.
Output: Drizzle table + migration 0025 (local-pushed), the `ChatWidgetInteractionRepository` port +
Supabase adapter, and the `validate_result_against_schema` domain service.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → FastAPI submit (next plan) | untrusted structured result crosses here; this plan stores the schema it will be checked against |
| DB row → server | a stale/tampered interaction row must never grant a second submit |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-01 | Tampering | client-supplied result vs declared contract | mitigate | The declared response schema is STORED in chat_widget_interactions at emit time; re-validation (Task 3) always reads the stored schema, never a client-supplied one (D-10) |
| T-24-02 | Tampering | double-submit / replay | mitigate | try_submit is a DB-level CAS (`WHERE state='pending'`); a second submit updates zero rows and returns False (D-11) |
| T-24-03 | Elevation of Privilege | anon/authenticated direct table access | mitigate | RESTRICTIVE deny-all RLS for anon + authenticated (mirrors 0023); only service_role/postgres backends touch the table |
| T-24-04 | Tampering | prototype-pollution / oversized jsonb in declaration or result | accept (this plan) | declaration/result jsonb are written by the trusted server from validated tool args (24-02) or validated at the submit boundary (24-02); no client writes this table directly |
| T-24-SC | Supply chain | new package installs | mitigate | none introduced — jsonschema + drizzle + supabase-py already in the repo; if any new package appears, stop and gate it |
</threat_model>
