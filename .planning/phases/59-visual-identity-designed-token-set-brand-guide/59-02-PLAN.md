## B. Archivo — the one live decision, with a hard fallback rule

58-IDENTITY.md does NOT name a typeface; its ladder has no font entries. Archivo is sketch
mechanics, therefore Claude's discretion. The sketch's own stack degrades gracefully to system
sans if Archivo is absent.

Decision rule (follow it exactly, do not deliberate at runtime):
1. Write `--font-sans` as the Archivo-first stack above. This is correct whether or not the
   webfont loads.
2. Attempt to self-host Archivo via `next/font/google` in `apps/web/src/app/layout.tsx`
   (weights 400 and 600 — the only two the sketch uses), exposing it as a CSS variable that
   `--font-sans` consumes.
3. **If `npm run build` fails to fetch the font** (no network — the overnight reality this repo
   plans around), REMOVE the `next/font` import, keep the Archivo-first stack unchanged, and
   record it in the SUMMARY as an honest documented follow-up (the 51-07 / 55-01 precedent).
   The identity does not depend on it; a red build does. Never leave the build red for a font.

## C. Tailwind v4 registration shapes (this repo's existing conventions)

- `@theme` (native, NOT inline) is where non-colour families register — the existing block
  already holds `--radius-*`, `--shadow-elevation-*`, `--font-sans`, `--font-mono`, `--font-code`.
  Add `--font-serif` and the `--text-*` scale there.
- `--text-<name>` registers a `text-<name>` utility; a paired `--text-<name>--line-height`
  registers its leading.
- `--spacing` sets the spacing-scale base; `--spacing-<name>` registers named steps.
- `@utility <name> { ... }` declares a custom utility — the file already uses this shape for
  `container`, `animate-in`, `touch-target`, etc. Follow it.
- Gotcha (carried from 59-01): `readTokenBlock(css, "@theme")` matches `@theme {` and NOT
  `@theme inline {`, and its regex is non-greedy to the first `}`. Keep the `@theme` block
  brace-flat and do not reorder the two blocks.

## D. Law-1 chroma separation — verified, the gate is clean

Computed across the ported ladder during planning:
- Max chroma on any CHROME token: **0.026** (`--rule`, dark).
- Min chroma on any EARNED-HUE token: **0.068** (`--conf`).
There is a wide, unambiguous gap. A ceiling of **0.03** for chrome and a floor of **0.06** for
the earned hues separates them cleanly with no borderline case. That is what makes law 1
gateable rather than aspirational.

Earned-hue set (the ONLY tokens allowed chroma > 0.03), closed list:
`conf`, `conf-wash`, `conf-line`, `sugg`, `sugg-wash`, `sugg-line`, `bad`, `bad-hi`,
`destructive`, `success`, `tier-extracted`, `tier-inferred`, and `chart-1..5`
(the documented out-of-contract exemption — see 59-01 interfaces §B).
Everything else in `:root`/`.dark` is chrome and must sit at or below 0.03.
