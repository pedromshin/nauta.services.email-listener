## B. What is already ENFORCED vs what the guide must carry alone

Do not pad the guide with rules a gate already owns — say which gate owns them and move on. The
guide's real job is the rules no gate can check.

ENFORCED by a committed gate (cite the gate, do not re-argue it):
- WCAG-AA on every semantic pair, the pencil/faded ground rules, and the tier-on-wash worst case
  -> `apps/web/src/app/__tests__/token-contrast.test.ts`
- Law 1 (no hue on chrome) + earned-hue floor + cross-theme hue/chroma invariance
  -> `apps/web/src/app/__tests__/colour-law.test.ts`
- Token families registered -> `apps/web/src/app/__tests__/token-registration.test.ts`
- No Tailwind palette classes in app source -> `apps/web/src/app/__tests__/palette-ban.test.ts`

NOT gateable — the guide is the only enforcement, so these must be stated crisply:
- Law 2's serif rule: the serif marks the USER'S OWN MATERIAL (mail, saved sources, values pulled
  out of them). Nothing polytoken says in its own voice wears it — NO exceptions, because one
  exception makes the rule unlearnable. (D-58-01 audited all 86 serif elements in the sketch;
  the only violation found was the sketch's own manifesto lede, which was set back to sans.)
- Law 3's placement rule: type shapes appear only where there is no room for a word (filter
  rails, canvas nodes), never beside a label that already says the type. There is deliberately
  no shape for `date`.
- Madder (`--bad`) is for the IRREVERSIBLE only. Never errors, never warnings.
- `--pencil` is legal on `--shelf`/`--leaf`/`--bright` but NEVER on `--shade` (4.23:1 — below AA).
  The gate catches the pair it can see; this rule stops the usage that produces it.
- Adding a hue requires a demonstrated usability failure of the monochrome treatment — not a
  preference — documented as an amendment to 58-IDENTITY.md.

## C. The two open flags — record them, do not resolve them

Both are real and both must appear in the guide (§5 "NOT done — user-gated" is the existing home
for user-gated items; use it or an equivalent subsection):

1. **D-58-03 (entity type is shape, never hue)** — 58-IDENTITY.md flags this as "the one item the
   user has not explicitly blessed"; it was inferred from the user's colour law, not instructed.
   59-CONTEXT.md's Deferred Ideas says to surface it IF the port makes the cost concrete. It did:
   law 3 forced `--graph-entity`/`--graph-email-component`/`--graph-email` to surrender their
   hues (59-01), and `extraction-summary-panel.tsx`'s `candidate: "bg-graph-email-component"`
   shows a surface that was using a node-type hue to mean a TIER. Cheap to revisit now, expensive
   after Phase 62. Record the concrete cost; laws 1 and 2 stand regardless.
2. **`--chart-1..5`** — the single family left out of the identity (59-01 interfaces §B): not in
   D-58-01's ladder, consumed only by the spreadsheet grid's conditional formatting as
   USER-ASSIGNED cell annotation, same category as the out-of-scope `packs.ts`. Left byte-identical
   and exempted by name in the law-1 gate. Needs a user decision: fold into the identity, or keep
   as a bounded user-annotation exemption. Also note the pre-existing, out-of-scope defect found
   during planning: `conditional-formatting-dialog.tsx` offers `chart-6`/`chart-7`/`chart-8`,
   which globals.css has never defined.
