## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| inbound email -> DB -> UI | `contentText` is attacker-authored text from an arbitrary sender; this plan puts it on a new surface (the chip) |
| client -> tRPC `entitySummary` | caller-supplied `emailIds` may include ids the caller does not own |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-60-01 | Information Disclosure | `entity-summary.ts` | mitigate | The new `value` field is email BODY content — strictly more sensitive than a type label. Keep `inArray(EmailComponents.importerId, owned)` in the where-clause verbatim; a foreign emailId must continue to yield `{ emailId, entities: [] }`. Do not widen the select beyond the already-filtered rows. |
| T-60-02 | Tampering (XSS) | `entity-chips.tsx` | mitigate | Render `value` as a React text node only (auto-escaped). Never `dangerouslySetInnerHTML`. Never interpolate `value` into an href, a `style`, or a class string. |
| T-60-03 | Denial of Service | `entity-summary.ts` | mitigate | Moving from a distinct-type rollup to per-entity entries removes the natural collapse: an email with hundreds of OCR entity regions x 100 emails could balloon the payload. Cap entries per email at `MAX_ENTITIES_PER_EMAIL = 8` and return `totalCount` so the client can render an overflow chip without the full list. |
| T-60-SC | Tampering | npm/pip/cargo installs | mitigate | Not applicable — this plan installs no packages. |

    THE CENTRAL DESIGN CONSTRAINT, and the reason this file exists: the fingerprint MUST be blind
    to colour. Criterion 1 says the inbox must differ "not just in color". If the fingerprint could
    see colour, a re-token would satisfy it and the gate would be theatre. Therefore
    `fingerprintTree` reads NO className, NO style, and NO `data-*` attribute. Excluding className
    makes a pure re-token invisible to it. Excluding `data-*` closes the loophole where a later plan
    "passes" the gate by sprinkling marker attributes instead of restructuring. What remains is pure
    DOM topology, which only genuine restructuring can move.

    `StructuralFingerprint` fields:
      - `shape: string` — a deterministic serialization of the subtree. Per element, emit its
        lowercased tagName; then `[role=VALUE]` if a role attribute exists; then, for the fixed
        allowlist `aria-expanded`, `aria-pressed`, `aria-selected`, `aria-hidden`, a
        PRESENCE-ONLY marker (never the value — values flip with state and would make the
        artifact unstable); then a `#t` marker if the element has a direct non-empty text child
        (the marker only, never the text — fixture text must not leak into the artifact); then its
        children recursively in document order. Use an unambiguous nesting syntax such as
        parentheses. Deliberately absent: className, style, data-*, id, href, aria VALUES.
      - `elementCount: number` — total elements in the subtree.
      - `maxDepth: number` — deepest nesting level.
      - `leafTextCount: number` — count of elements with a direct non-empty text child. This is the
        information-density proxy: it counts distinct rendered facts, not pixels, which is what
        "information density" means in ROADMAP criterion 2 and what jsdom can honestly measure
        (jsdom does not evaluate Tailwind CSS, so a real px density metric is NOT available here —
        that is the screenshot harness's job in Plan 05).
      - `tagCounts: Record<string, number>` and `roleCounts: Record<string, number>` — sorted-key
        maps, for diagnosing WHICH structures were added when the gate reports a delta.
