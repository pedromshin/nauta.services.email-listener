<objective>
Close criterion 4 — and backlog 999.17's read half: the docked/mobile transcript reflects panel
overlays and rethemes made on the canvas side of the same conversation.

Purpose: a user re-themes a panel on the canvas, switches to Chat, and sees the ORIGINAL. Two views
of one conversation disagreeing about what a panel looks like — filed as 999.17 and carried as a
STATE.md `integration_warning` since Phase 52. The cause is structural, not a missing feature:
`usePanelOverlay` needs `CanvasStoreProvider` AND `CanvasPersistenceProvider`, and **only
`chat-canvas.tsx` provides them**, so the transcript is architecturally blind to the overlay store.
Everything else is already in place — `genuiPanelNodeId` and `resolveActivePanel` are pure,
`chat.getCanvasLayout` is a plain query, and `PanelThemeScope` is a component. This plan supplies
the missing seam.

**It also carries this phase's one genuine data-loss hazard** (T-61-21). Read it before writing a
line: a naive host wipes the user's entire canvas layout on the first mobile edit.

Output: a `TranscriptPanelHost`, a non-throwing overlay read, a transcript that resolves overlays,
and a regression test that fails if a transcript save ever empties the layout.
</objective>

  export function useCanvasPersistenceContext(): CanvasPersistenceContextValue {
    const ctx = useContext(CanvasPersistenceContext);
    if (ctx === null) throw new Error(
      "usePanelOverlay must be used inside a CanvasPersistenceProvider (canvas host wiring — see chat-canvas.tsx)");
    return ctx;
  }
  export function usePanelOverlay(panelId: string): UsePanelOverlayResult {
    const store = useCanvasStore();                      // throws without CanvasStoreProvider
    const { scheduleSave } = useCanvasPersistenceContext();   // throws without CanvasPersistenceProvider
    const rawOverlay = useStore(store, useShallow((state) =>
      resolveCanvasPath(state.values, `shared.panelOverlays.${panelId}`)));
    const overlay = useMemo(() => parseOverlay(rawOverlay), [rawOverlay]);   // memo on the RAW ref — see §F
    const mutate = useStore(store, (state) => state.mutate);
    const writeOverlay = useCallback((next, onSaveError?) => {
      mutate("set", `shared.panelOverlays.${panelId}`, next);
      scheduleSave(onSaveError);
    }, [mutate, panelId, scheduleSave]);
    return { overlay, writeOverlay };
  }

  interface CanvasPersistenceContextValue {
    readonly scheduleSave: (onError?: () => void) => void;   // pre-bound to the store by chat-canvas.tsx
    readonly conversationId: string;
    readonly onOpenConversation?: (conversationId: string) => void;
  }

§D — **THE THREE-TREE PROBLEM.** `MessageList` -> `MessageTurn` renders in THREE different trees:

  1. DOCKED (page.tsx's chat branch)  — no canvas providers today. This plan adds TranscriptPanelHost.
  2. ON CANVAS (chat-node.tsx's ChatNodeBody) — **already inside chat-canvas.tsx's full provider
     stack.** A ChatNode is a node ON the canvas whose body is a MessageList.
  3. BARE (unit tests / any future mount) — no providers at all.

  So an overlay read from `MessageTurn` MUST be NON-THROWING, or case 2 keeps working by accident
  and case 3 breaks every existing chat suite. The codebase already has the exact precedent, in the
  file next door (chat-node.tsx:90):
    export function useOptionalChatController(): ConversationController | null {
      return useContext(ChatControllerContext);   // null when no provider — never throws
    }
  Its doc explains the split: a MISSING provider is a host-wiring bug for a component that can only
  legitimately exist inside the host (throw), but a degraded/standalone mount of a shared component
  is a real case (return null). `MessageTurn` is the second kind. Mirror it.

  In case 2, do NOT mount a second host: the canvas's providers are already there, so the same
  optional read resolves against the canvas's own live store and the ChatNode transcript starts
  honoring overlays for free. **Never nest TranscriptPanelHost inside the canvas** — two stores for
  one conversation is the drift this whole plan exists to end.

§E — **T-61-21: THE LAYOUT-DESTRUCTION HAZARD. READ THIS BEFORE WRITING THE HOST.**

  use-canvas-persistence.ts:303-347 —
    export interface UseCanvasPersistenceOptions {
      readonly conversationId: string;
      readonly nodes: readonly FlowNode[];      // LIVE React Flow state
      readonly edges: readonly FlowEdge[];
      readonly viewport: Viewport | null;
    }
  and scheduleSave's timer (line 463-500) —
    const current = latestStateRef.current;                                  // = {nodes, edges, viewport}
    const sharedState = canvasStoreRef.current?.getState().values ?? {};
    snapshot = buildSnapshot(current.nodes, current.edges, current.viewport, sharedState);
    saveMutation.mutate({ conversationId, snapshot });                       // UPSERTS THE WHOLE ROW

  `chat.saveCanvasLayout` **upserts by conversationId — one row, full replace.** `CanvasSnapshotSchema`
  requires `nodes` and `edges`; there is no partial-save path.

  **So a host that calls `useCanvasPersistence({conversationId, nodes: [], edges: [], viewport: null})`
  and then lets a panel write an overlay will persist `nodes: []` — silently deleting every node
  and edge the user has ever placed on that conversation's canvas, from a phone, on a re-theme.**
  It would pass every existing test, because no test asserts that a save preserves the layout.

  THE CONSTRAINT (the mechanism is yours; this is not):
    **A save scheduled from the transcript must round-trip the persisted layout unchanged.**
  The restore side already hands you exactly what you need — `useCanvasPersistence` returns
  `initialNodes: ReconciledNode[]`, `initialEdges`, `initialViewport`, `initialSharedState` — so
  feeding those back as the live `nodes`/`edges`/`viewport` makes a transcript save write back what
  it read, plus the new `sharedState`. Note the shape gap: `ReconciledNode` is
  `{id, type, position, data, isNew}` (use-canvas-persistence.ts:69) while `nodes` wants `FlowNode`;
  `toFlowNode` (chat-canvas.tsx:177) does that conversion and is currently **module-private**.
  Export it or move it — do not write a second copy.

  Do NOT gate on `isRestoring`: writing an overlay before the layout has restored is the same bug
  wearing a race. The host must not provide a persistence context until it has a real snapshot.

  And a genuinely-new panel (a turn whose node the canvas has not placed yet) simply has no node row
  to preserve — writing `shared.panelOverlays.{panelId}` for it is correct and harmless: the canvas
  reconciles the node from `historyRows` on its next mount and finds the overlay waiting.
