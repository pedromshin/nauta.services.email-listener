<objective>
Give the canvas a tier/kind vocabulary that is GROWN from Phase 60's, not improvised beside it — and
promote the tier truth to a home three surfaces can share without importing each other's internals.

Purpose: `region-vocabulary.ts` made the single most reusable decision Phase 60 made — **tier owns
colour and solid-vs-dashed; role owns weight, style and geometry, never hue** — and
`docs/design/brand-guide.md` §3 says outright that Phases 61-63 need it *for canvas nodes and
edges*. Right now the canvas answers the same questions with five improvised local class maps and a
retired node-type hue family, which is precisely how the `graph-*` debt accumulated in the first
place. This plan builds the canvas's half of the vocabulary and moves the shared fact — "what tier
is this?" — somewhere `/chat`, `/emails/[id]` and (next phase) `/knowledge` can all read without a
cross-surface import.

Output: a shared `_vocabulary/tier.ts`, a `canvas-vocabulary.ts` grown from it, and a matrix gate
over the full edge-tier × node-kind space. No component changes — Plans 61-04 and 61-05 wire them.
</objective>

§B — The rule being generalized (brand-guide.md §3, VERBATIM — do not re-derive it):

  > Tier owns colour and solid-vs-dashed. Role owns weight, style, and opacity — never hue.
  > The two axes are orthogonal: a box reads "entity, suggested" as *thick and amber-dashed*, and
  > neither reading interferes with the other.

  Plus: "One mapping, not two. Do not re-derive tier locally next to a component that already has
  it — two maps of one fact drift, and the drift reads to the user as two panels disagreeing."

§C — WHY the shared module holds FACTS, not classes (the constraint that shapes Task 1):

  Tailwind v4 scans SOURCE FILES for literal class strings (globals.css's `@source` directives —
  55-02 already ate one purge bug from this). A composed class (`` `border-${family}-line` ``) is
  invisible to the scanner and is silently dropped at build time with no error. So per-surface
  LITERAL class maps are unavoidable; they cannot be generated from a shared token name.

  Therefore the shared module owns the tier TRUTH and the FACTS each surface's literal map must
  agree with, and each surface's gate asserts its own literals against those facts. That is how you
  get one truth AND literal classes:

    export type Tier = "confirmed" | "suggested" | "terminal";
    export function tierOf(status: string): Tier
    export const TIER_HUE_FAMILY: Record<Tier, "conf" | "sugg" | null>
      // confirmed -> "conf", suggested -> "sugg", terminal -> null (NO tier claim, so no colour)
    export const TIER_IS_DASHED: Record<Tier, boolean>
      // confirmed -> false (SOLID mark), suggested -> true (DASHED mark), terminal -> true
