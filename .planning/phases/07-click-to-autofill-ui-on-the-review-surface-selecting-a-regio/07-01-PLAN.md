The new mutations are added to the EXISTING `componentMutationProcedures` object
in mutations.ts (a plain object spread into `emailsRouter` in
`packages/api-client/src/router/emails/index.ts`). They are therefore reached by
the UI as `api.emails.autofillComponent`, `api.emails.confirmComponent`,
`api.emails.reprocessEmail` — matching every `api.emails.*` reference in 07-UI-SPEC.

autofillComponent — input { componentId: uuid, entityTypeSlug: string(min 1) }
  → POST {url}/v1/components/{componentId}/autofill  body { entity_type_slug }
  → returns res.json() as Promise<unknown> (FastAPI wraps in { data, ok })

confirmComponent — input { componentId: uuid, correctedFields: Record<string,unknown> | null }
  → POST {url}/v1/components/{componentId}/confirm  body { corrected_fields: correctedFields }
  → returns res.json() as Promise<unknown>

reprocessEmail — input { emailId: uuid }
  → POST {url}/v1/emails/{emailId}/reprocess  body {}
  → returns res.json() as Promise<unknown>

entityTypesRouter.list — input z.object({}).optional()
  → Drizzle read; output Array<{ slug, label, description, fields: Array<{ key, label, dataType, isRequired }> }>

emails.detail component select — ADD three columns (correctedFields,
  confidenceBreakdown, extractionRecordStatus); extractedFields/confidenceScore/
  entityTypeSlug already present.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| web client → tRPC (server) | Browser-originated mutation inputs cross into server-side procedures |
| tRPC (server) → FastAPI | Server-side fetch carries EMAIL_LISTENER_API_KEY to the email-listener service |
| FastAPI → Drizzle/Supabase | Out of scope this phase (backend shipped Phase 4, not modified) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Information Disclosure | EMAIL_LISTENER_API_KEY | mitigate | Key read only inside `getListenerConfig()` at call time (existing T-06-10 idiom); never `NEXT_PUBLIC_`. Grep gate asserts no `NEXT_PUBLIC_EMAIL_LISTENER` token in mutations.ts. |
| T-07-02 | Tampering | componentId / emailId path params | mitigate | `z.string().uuid()` validates before any URL interpolation, preventing path-segment injection into the FastAPI URL. |
| T-07-03 | Tampering | correctedFields user input | accept | Passed through as JSON to FastAPI which validates and persists (Phase 4 contract). The web proxy never eval's or renders it; it is opaque `Record<string,unknown>`. Rendering happens read-only in 07-03 via React text nodes (auto-escaped). |
| T-07-04 | Information Disclosure | FastAPI error bodies | mitigate | Only `parseErrorDetail`'s `{detail}` string is surfaced; no raw body/stack echoed (existing idiom). |
| T-07-05 | Denial of Service | reprocessEmail (unauthenticated, rate-unlimited) | accept | Single-key internal tool, D-18 (X-API-Key = installation-wide principal, no per-user auth yet). Rate limiting is a future auth-seam concern; documented accepted risk. |
| T-07-SC | Tampering | npm installs | accept | No new packages installed this plan (Component Inventory §4 of UI-SPEC confirms all deps already present). No legitimacy gate needed. |
</threat_model>
