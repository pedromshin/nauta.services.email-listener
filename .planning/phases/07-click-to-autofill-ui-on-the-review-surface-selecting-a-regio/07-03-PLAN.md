<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| extracted field values → DOM | extractedFields / correctedFields values rendered in inputs (reviewing) and text (confirmed) |
| web client → tRPC (server) | confirmFields + reprocess inputs cross into server-side proxy |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-09 | Spoofing/XSS | extracted/corrected field values | mitigate | Values rendered via controlled `<Input value>` and read-only `<p>{value}</p>` — React auto-escapes; no `dangerouslySetInnerHTML`. JSON values coerced with `String(v ?? "")`. |
| T-07-10 | Repudiation | reprocess (supersede-safe) | mitigate | Reprocess gated behind an AlertDialog requiring explicit confirm (mirrors T-06-17); copy states confirmed data is never deleted (D-16). |
| T-07-11 | Tampering | confirm corrected_fields payload | accept | Server (Phase 4) validates + persists; the UI sends only the diff keys, server is source of truth (invalidate after success — no optimistic). |
| T-07-12 | Information Disclosure | confirm/reprocess failures | mitigate | Only fixed friendly toasts shown ("Could not confirm fields. Try again.", "Could not reprocess email. Try again."); raw errors never rendered. |
| T-07-SC | Tampering | npm installs | accept | No new packages (Input/Badge/AlertDialog/Loader2 already present — UI-SPEC §4). No legitimacy gate needed. |
</threat_model>
