<objective>
Make the canvas persistent and streaming-responsive (CANVAS-02, CANVAS-04, D-06/D-07). On mount, fetch `chat.getCanvasLayout` and restore saved node positions/edges/viewport exactly (`toObject()`-style), reconciling with `chat.getHistory` so existing panels keep their saved positions, new genui parts get dagre/offset-cascade placement, and any node whose type is unknown to the current `NODE_TYPE_REGISTRY` degrades to the inert placeholder (honoring its saved position, never blanking the canvas). Save a debounced (~800ms), coalesced snapshot on drag/edge/viewport settle via `chat.saveCanvasLayout` (positions/edges/viewport/shared_state + `NODE_REGISTRY_VERSION`, NO spec content), with an ambient `SaveStatusIndicator`. Finally, wire streaming so volatile content flows through `CanvasSpecContext` (fed by the controller's `useChatStream`) INSIDE the memoized nodes — a streamed token never calls `setNodes` and never relayouts the graph.

Purpose: This closes the persistence loop opened by 23-01/23-03 and delivers the requirement-named responsiveness property (volatile state outside the `nodes` array, memoized node types).

Output: `useCanvasPersistence` (restore + debounced save + reconcile), `SaveStatusIndicator`, and the streaming-override wiring into `CanvasSpecContext`.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| persisted layout row → canvas restore | A tampered/legacy snapshot (unknown types, stale registry version) is applied on mount |
| streamed token flow → React state | High-frequency updates could thrash the graph |
| debounced save → tRPC | Repeated user edits pushed to persistence |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-23-09 | Tampering | restored layout with unknown types / stale node_registry_version | mitigate | Per-node degrade to inert placeholder (position honored); never throws or blanks the canvas (CANVAS-03 carry-through); restore validated against CanvasSnapshotSchema on the read side too |
| T-23-10 | Denial of service | save spam / streamed-token relayout thrash | mitigate | Debounce ~800ms single trailing timer coalesces edits; streamed content flows through context (never setNodes per token); node dims fixed during streaming |
| T-23-02 | Tampering | spec content persisted into the saved snapshot | mitigate | `buildSnapshot` stamps provenance refs only + asserts no `spec`/`root`; CanvasSnapshotSchema re-validates at the tRPC boundary (23-01) |
| T-23-SC | Tampering | npm/pip/cargo installs | accept | No package installs; zustand install gated in 23-05 |
