Close the Phase 23 verification gap on success criterion #5 (STATE-01/STATE-02): the canvas
store's WRITE path has zero production call sites — `usePanelData().dispatch` / store `mutate`
is invoked only from its own test file, so `panels.*`/`shared.*` stay `{}` forever,
`EdgeCreationPicker`'s source-field list is always empty, and no user can create or observe a
working data-carrying edge end-to-end.

**The bridge (two halves, neither touches SpecRenderer):**

1. **Trigger half (packages/genui catalog):** `ButtonComponent` in `catalog/manifest.ts` today
   IGNORES its validated `onClick`/`action` props — clicks are inert everywhere. This completes
   the package's OWN documented design: `spec-renderer.tsx`'s header (lines 8-11) literally states
   "button onClick will call useActionRegistry(action) ... Real handlers are registered via
   <ActionRegistryContext.Provider>", and `FormComponent` (form-component.tsx:207,235) already
   implements the exact consumption pattern (`registry[action.type]?.(action)` in try/catch).
   Wire ButtonComponent to that same context. `spec-renderer.tsx` itself is NOT modified.

2. **Handler half (apps/web canvas layer):** a new `panel-action-bridge.ts` builds a per-panel
   `ActionRegistry` whose ONLY entry is `setState`, routing writes through the existing bounded
   write surface: plain keys -> `usePanelData().dispatch("set", key, value)` (lands at
   `panels.{panelId}.{key}`), `shared.`-prefixed keys -> the store's `mutate("set", path, value)`
   (lands at `shared.{key}`). `GenuiPanelNodeBody` stops discarding `dispatch` and threads the
   registry through a new additive `actions` prop on `GenuiPartBoundary` into the UNMODIFIED
   `SpecRenderer`'s existing `actions` prop.

picker's source-field list becomes non-empty, and an edge from panel A's output feeds panel B's
data live.
