## Entity Chip href Flip (Phase 9 D-24 → Phase 10 activation)

**Decision source: 09-CONTEXT.md D-24; 10-CONTEXT.md code_context.**

`apps/web/src/app/_components/entity-chips.tsx` currently computes:
```tsx
const href = `/emails/${emailId}`;
```

Phase 10 changes this to link each chip to the entity's own page. The chip receives the `entityInstanceId` alongside `emailId`. The updated prop contract:

```tsx
export interface EntityChipEntry {
  readonly entityTypeId: string;
  readonly label: string;
  readonly count: number;
  readonly entityInstanceId: string;   // NEW — added by Phase 10 entityInstances resolution
}
```

Href computation:
```tsx
const href = `/entities/${entity.entityInstanceId}`;
```

The `emailId` prop is no longer needed on `EntityChipsProps` once all chips have `entityInstanceId`. During the transition (entity resolution partially populated), chips without an `entityInstanceId` fall back to `/emails/${emailId}` — the executor should handle this gracefully with a conditional in the href expression.

### Curation Confirmation Dialogs (D-20 — AlertDialog for destructive/irreversible)

All three curation actions that are irreversible use `AlertDialog` (`@nauta/ui/alert-dialog`). Confirm merge is NON-destructive (it creates an alias and links occurrences) and does NOT require a dialog — it fires directly with optimistic update. The destructive pair (reject, unmerge) require dialogs.

#### Reject Merge Suggestion Dialog

Trigger: "Reject" button on a suggestion row sets `rejectTarget` state → opens dialog.

```
AlertDialogTitle:   "Reject this merge suggestion?"
AlertDialogDescription:
  "This suggestion will be dismissed. The two entities will not be linked.
   You can still merge them manually later if needed."
Cancel button:      "Keep suggestion"   (AlertDialogCancel, default variant)
Confirm button:     "Reject suggestion" (AlertDialogAction, destructive variant)
```

`aria-label` on the confirm action: `"Confirm rejection of merge suggestion"`.

#### Unmerge Dialog

Trigger: "Unmerge" button in detail header sets `unmergeDialogOpen`.

```
AlertDialogTitle:   "Unmerge this entity?"
AlertDialogDescription:
  "This will separate the merged entities back into distinct records.
   Occurrences will be reassigned to their original entities.
   This action cannot be undone automatically."
Cancel button:      "Keep merged"  (AlertDialogCancel)
Confirm button:     "Unmerge"      (AlertDialogAction, destructive variant)
```

`aria-label` on the confirm action: `"Confirm entity unmerge"`.

### Optimistic Updates (D-21 — reuse `use-region-edit.ts` snapshot/revert idiom)

| Action | Optimistic behavior | Revert trigger |
|--------|---------------------|----------------|
| Confirm merge | Immediately move suggestion from "pending" list to a "merging…" spinner state; remove suggestion row on success | Error: restore row, toast error |
| Reject suggestion | Immediately remove suggestion row | Error: restore row, toast error |
| Unmerge | Immediately show "unmerging…" spinner in header; navigate away on success | Error: restore button state, toast error |

Optimistic state is managed via a snapshot of the current suggestions array before mutation, restored on error.

Bedrock degradation (D-12): if `entityInstances.byId` returns data without knowledge nodes (Bedrock 404 on embedding generation), the knowledge nodes section renders its empty state — NOT an error — since the entity record itself is lexical-arm populated and always exists. The page renders whatever is available.

## API + Mutation Patterns (D-21)

All writes route through the FastAPI proxy. Browser never holds `X-API-Key`. `importer_id` derived from the data row, never the caller (Phase 9 D-18 tenant isolation).

| Mutation | tRPC Procedure | FastAPI Endpoint | Trigger | Optimistic? |
|----------|---------------|-----------------|---------|-------------|
| Confirm merge suggestion | `entityInstances.confirmMerge` | `POST /v1/entity-instances/{id}/merge/{targetId}/confirm` | "Confirm merge" button | Yes — remove suggestion row, show spinner |
| Reject merge suggestion | `entityInstances.rejectMerge` | `POST /v1/entity-instances/{id}/merge/{targetId}/reject` | "Reject suggestion" → AlertDialog confirm | Yes — remove suggestion row |
| Unmerge entity | `entityInstances.unmerge` | `POST /v1/entity-instances/{id}/unmerge` | "Unmerge" → AlertDialog confirm | Yes — show spinner in header |

Queries:
- `entityInstances.list` — gallery, `limit+1` pagination, importer-scoped, supports `entity_type_id`, `status`, `search`, `sort` params (D-17)
- `entityInstances.byId` — detail page, returns entity + occurrences + field values + knowledge nodes + pending suggestions (D-18)

Flywheel write-back (D-11): the backend confirm-merge handler writes the variant spelling as an alias on the surviving entity; the frontend does not need to trigger this explicitly — it happens server-side.
