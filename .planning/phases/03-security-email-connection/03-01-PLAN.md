# Plan 03-01: SES Inbound Infrastructure

## Goal
Configure AWS SES to receive inbound email for `magnitudetech.com.br`, set the MX record so
mail routes to SES, and deliver each received email as an SNS HTTP POST to the running ALB.
After both plans are deployed, sending an email to `agent@magnitudetech.com.br` will result
in an SNS notification arriving at `/v1/emails/inbound-sns` on port 80 and a log line confirming receipt.

## Pre-conditions
- Terraform state is current (run `terraform plan` before apply; see Drift Fix section)
- DNS access to magnitudetech.com.br (user manages it)
- ALB DNS: `nauta-services-email-listener-2115368239.us-east-1.elb.amazonaws.com`
- AWS region: `us-east-1` (SES inbound receiving supported here)
- Check SES sandbox status: AWS Console → SES → Account Dashboard → Sending quota.
  If account is in sandbox, external email from unverified senders may be dropped silently.
  To exit sandbox: submit a support request in AWS Console → SES → Account Dashboard → Request production access.

## Terraform Drift Fix (do BEFORE any apply)
Before any `terraform apply`, import the drifted security group rule:

```bash
cd infrastructure/aws
terraform import aws_security_group_rule.alb_staging_port sgr-0791694527ab7b257
```

## Wave ordering note
Plan 03-01 (infrastructure) and Plan 03-02 (application code) must be coordinated:
- **Apply 03-01 first** to create SES domain identity + SNS topic + receipt rule
- **Deploy 03-02 first** to get the SNS confirmation handler live on the ALB
- **After 03-02 is deployed**, SNS will retry the SubscriptionConfirmation (it retries for up to 3 days)
- SNS subscription uses `endpoint_auto_confirms = false` so Terraform does not block waiting for confirmation

## Tasks

### Task 1 — SES domain identity + verification (`infrastructure/aws/ses.tf`)

Create `infrastructure/aws/ses.tf`:

```hcl
# SES domain identity for magnitudetech.com.br
resource "aws_ses_domain_identity" "main" {
  domain = "magnitudetech.com.br"
}

output "ses_domain_verification_token" {
  value       = aws_ses_domain_identity.main.verification_token
  description = "Add as TXT record: _amazonses.magnitudetech.com.br"
}
```

After `terraform apply`, DNS action:
- Add TXT record `_amazonses.magnitudetech.com.br` with the output token value
- Wait for SES to show Status: Verified (usually under 1 minute after DNS propagates)

### Task 2 — SNS topic for inbound email delivery

Add to `ses.tf`:

```hcl
resource "aws_sns_topic" "ses_inbound" {
  name = "${var.project}-ses-inbound"
}

resource "aws_sns_topic_policy" "ses_inbound" {
  arn = aws_sns_topic.ses_inbound.arn
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Sid       = "AllowSESPublish"
      Effect    = "Allow"
      Principal = { Service = "ses.amazonaws.com" }
      Action    = "SNS:Publish"
      Resource  = aws_sns_topic.ses_inbound.arn
    }]
  })
}
```

> Use `var.project` (not `local.project`). `local.project` is not defined in `locals.tf`.
> `var.project` defaults to `"nauta-services"` per `terraform.tfvars`.

### Task 3 — SNS HTTP subscription to ALB (unauthenticated SNS path)

Add to `ses.tf`:

```hcl
resource "aws_sns_topic_subscription" "alb_http" {
  topic_arn = aws_sns_topic.ses_inbound.arn
  protocol  = "http"
  # Uses /inbound-sns — the unauthenticated SNS-specific route (no X-API-Key required)
  endpoint               = "http://${var.alb_dns_name}/v1/emails/inbound-sns"
  endpoint_auto_confirms = false  # application must be deployed first; SNS retries for 3 days
}
```

Add to `infrastructure/aws/variables.tf` (create file if it does not exist):

```hcl
variable "alb_dns_name" {
  description = "ALB DNS name for SNS HTTP subscription"
  type        = string
}
```

Add to `terraform.tfvars`:

```hcl
alb_dns_name = "nauta-services-email-listener-2115368239.us-east-1.elb.amazonaws.com"
```

### Task 4 — SES receipt rule set + catch-all receipt rule

Add to `ses.tf`:

```hcl
resource "aws_ses_receipt_rule_set" "main" {
  rule_set_name = "${var.project}-inbound"
}

resource "aws_ses_active_receipt_rule_set" "main" {
  rule_set_name = aws_ses_receipt_rule_set.main.rule_set_name
}

resource "aws_ses_receipt_rule" "catch_all" {
  name          = "catch-all"
  rule_set_name = aws_ses_receipt_rule_set.main.rule_set_name
  recipients    = []   # empty = catch-all for the verified domain
  enabled       = true
  scan_enabled  = true

  sns_action {
    topic_arn = aws_sns_topic.ses_inbound.arn
    position  = 1
    encoding  = "BASE64"
  }
}
```

### Task 5 — Apply and verify infrastructure

```bash
cd infrastructure/aws
terraform plan   # review ses_inbound resources; confirm no errors
terraform apply
```

Verify in AWS Console:
- SES → Domains → magnitudetech.com.br → Status: Verified (after DNS TXT record added)
- SES → Rule Sets → `nauta-services-inbound` rule set is active, catch-all rule enabled
- SNS → Topics → `nauta-services-ses-inbound` topic exists
- SNS → Subscriptions → http subscription Status: **PendingConfirmation**
  (expected at this stage — will become Confirmed after 03-02 is deployed)

### Task 6 — MX record change (manual DNS step)

After SES domain is verified and receipt rule is active, update DNS at the registrar:

| Name                 | Type | Priority | Value                                     |
|----------------------|------|----------|-------------------------------------------|
| magnitudetech.com.br | MX   | 10       | inbound-smtp.us-east-1.amazonaws.com      |

**Priority guidance:** A lower MX priority number = higher preference. To ensure SES receives
the mail before Google Workspace, set the SES MX at a **lower number** than Google's records.
Example: if Google's MX records are at priority 10 and 20, add SES at priority **5**.

**Safest PoC option (no MX change risk):** Configure a Gmail forwarding filter on
`agent@magnitudetech.com.br` (Google Workspace admin → Gmail → routing → or user-level filter)
to forward all mail to the ALB URL. SNS is not involved in this path — the Gmail forward would
need to POST to the endpoint, which requires a webhook-compatible Gmail forwarding setup.
The MX change + SES path is more reliable for production architecture.

- `Type: Notification` → `Message` contains JSON with raw email metadata (BASE64 encoding: raw email is base64 in `content` field)
