<objective>
Make the read-only `EntityTypeRepository` write-capable and expose entity-type/field CRUD through a NEW FastAPI router `/v1/entity-types` (D-26), with validation + integrity guards (D-27): `field_type` constrained to `{string,number,date,array,object}`, `slug` uniqueness per entity type, and a delete-guard that blocks (or soft-deactivates) deleting a field still referenced by a confirmed component's `entity_type_field_id` (the D-04 FK). System-default entity types (`importer_id` NULL) are editable; per-importer overrides stay out of scope. No new tables — `entity_types` + `entity_type_fields` already exist and are seeded with 8 system types.

Purpose: This is the backend the `/entity-types` management page (09-07) and the tRPC write router (09-04) depend on. It is independent of 09-02 (different files: entity_types.py vs components.py) so the two backend plans run in parallel.

Output: extended EntityTypeRepository (port + Supabase impl) with write methods + a reference-count query for the delete-guard, a `manage_entity_types.py` use-case module, a new `/v1/entity-types` router mounted in the app, DI registrations, and fake-repo unit tests covering validation + the delete-guard.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> tRPC -> FastAPI /v1/entity-types | NEW write surface; entity-type schema is shared installation config. |
| field delete -> confirmed components | Deleting a referenced field could orphan email_components.entity_type_field_id (D-04 FK). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-20 | Spoofing | /v1/entity-types write router | mitigate | Router-level `dependencies=[Depends(require_api_key)]` — all create/update/delete/reorder endpoints require X-API-Key. |
| T-09-21 | Tampering | field_type free-text injection | mitigate | Pydantic field_validator restricts field_type to {string,number,date,array,object} (D-27); use case re-validates against ALLOWED_FIELD_TYPES (defense in depth). |
| T-09-22 | Tampering | duplicate slug corrupting lookups | mitigate | DB UNIQUE(importer_id, slug) + per-entity-type field-slug check in the use case; conflict -> 409, not a silent overwrite. |
| T-09-23 | Denial of Service / data loss | orphaned entity_type_field_id on field delete | mitigate | DeleteFieldUseCase calls count_confirmed_references; references > 0 -> soft-deactivate/block (never hard-delete), preserving the D-04 FK integrity. |
| T-09-24 | Information Disclosure | error detail leakage | mitigate | Generic 404/409 detail to the client; full context logged server-side (structlog). |
| T-09-SC | Tampering | npm/pip/cargo installs | mitigate | No new Python packages (fastapi/pydantic/dishka/supabase already present); legitimacy gate N/A. |
</threat_model>
