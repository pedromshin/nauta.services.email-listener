<objective>
Deliver the safe agent↔user round-trip backend: a new `emit_proposal_cards` tool the agent may call
to end its turn with a pending widget, and a `POST /v1/chat/widget/submit` endpoint that enforces
DCUI-03's guarantees — re-validate against the STORED schema, DB-level double-submit lock,
turn-bound staleness, explicit-action-only — before persisting the structured result and resuming
the run as a streamed continuation turn (async-resume, no held-open stream, D-01).

Purpose: this is the reusable round-trip machinery. The submit endpoint is generic over widget kind
(24-04's clarify-widget reuses it unchanged); this plan wires the proposal-card emit path end-to-end
on the server so 24-03 can make it observable in the UI.
Output: the emit tool, the run-loop finalization + pending-row creation, the `SubmitWidgetInteraction`
use case, the SSE submit endpoint, and DI/router wiring.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Next proxy → FastAPI submit | untrusted structured result + interactionId cross here |
| model output → tool args | untrusted model-authored widget declaration crosses into persistence |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-01 | Tampering | client submits an arbitrary payload | mitigate | Client submits only an optionId (proposal) or values validated against the STORED schema (clarify); the server resolves the real payload from the stored declaration — arbitrary client payloads are never trusted (D-05/D-10) |
| T-24-02 | Tampering | double-submit race | mitigate | SubmitWidgetInteraction gates the resume on try_submit (24-01 CAS); conflict → 409, run not resumed (D-11) |
| T-24-03 | Tampering | stale/superseded submit resumes an outdated run | mitigate | is_stale check runs before the lock; stale → 409, run not resumed (D-12) |
| T-24-04 | Spoofing | submit references another conversation's interaction | mitigate | The use case verifies interaction.conversation_id == body.conversation_id (404 otherwise); X-API-Key auth on the router (two-hop key) |
| T-24-05 | Injection | validated result flows into model context | mitigate | Result is schema-validated then attributed as a clearly-delimited synthesized user turn (interaction_result stand-in); no tool_use/tool_result block smuggling |
| T-24-06 | Denial of Service | auto-fire / no explicit action | mitigate | Nothing resumes without an explicit POST carrying a user-chosen optionId/values; the emit tool ends the turn and waits (never auto-fires) |
| T-24-07 | Tampering | prototype pollution via tool-arg / result keys | mitigate | Tool input_schema is additionalProperties:false; the widget spec still passes SpecRootSchema.safeParse at the web boundary (24-03); result keys constrained by the declared schema |
| T-24-SC | Supply chain | new package installs | mitigate | none introduced this plan |
</threat_model>
