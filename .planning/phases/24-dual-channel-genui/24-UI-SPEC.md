# Phase 24 — UI Design Contract

> Visual and interaction contract for agent↔user widget round-trips — proposal cards, then
> clarify-widgets — rendered through the UNMODIFIED `SpecRenderer` + existing catalog/form-engine,
> in both the docked chat transcript and the canvas `genui-panel` node. Generated by
> gsd-ui-researcher (autonomous session — no user prompts; all choices derived from
> `24-CONTEXT.md`'s 16 locked decisions [D-01..D-16], the approved `22-UI-SPEC.md` chat contract,
> the approved `23-UI-SPEC.md` canvas contract, and `23-UI-REVIEW.md`'s live findings against the
> ACTUAL shipped code). Verified by gsd-ui-checker.

**Extends, does not replace:** `22-UI-SPEC.md`'s `MessageTurn`/interleaved-parts model (D-18) and
`GenuiPartBoundary`, and `23-UI-SPEC.md`'s `GenuiPanelNode` canvas chrome. No new brand colors,
radii, fonts, or catalog components are introduced anywhere in this spec — DCUI-01/02 are explicit
that proposal cards and clarify-widgets render via the EXISTING declarative catalog (card/button)
and the Phase-19 form engine only (D-05, D-09).

**Mandatory prerequisite this phase must land before/alongside new widget chrome:**
`23-UI-REVIEW.md` (2026-07-05, live code audit) found `GenuiPanelNode` triple-nests chrome — its
own node shell + an inner `p-4` div + `GenuiPartBoundary`'s own unconditional `GenuiCard` wrapper
stack three boxes around 320px-wide content. This phase adds a NEW state layer (pending/submitted/
superseded/stale badges + a submitting/error row) on top of that same boundary. Adding a fourth
layer on top of an already-broken three would make the regression worse and directly violates this
phase's own "no MORE nesting" scope hint. **Fix the boundary as part of this phase's build, not
after it** — see Component Inventory's `GenuiPartBoundary` `variant` prop.

## Layout & Structure

DCUI adds no new route/page. It adds **two new typed message-part variants** to the existing D-18
interleaved-parts model (`text` | `genui_spec` | `genui_spec_streaming` today), rendered inside the
SAME two hosts that already render `genui_spec` parts:

1. **`MessageTurn`** (docked chat transcript, `22-UI-SPEC.md`) — new part branches slot into the
   existing `parts.map` inside the same `space-y-2` container, same vertical rhythm as every other
   part.
2. **`GenuiPanelNode`** (canvas, `23-UI-SPEC.md`) — same part content, same underlying message-part
   data (D-08: "interactive state is shared via message parts, never duplicated") — whichever
   surface the user interacts on, both reflect the update immediately, exactly like Phase 23's
   shared conversation state already does for streaming text.

### New part shapes (visual contract only — exact JSON/DB shape is a planning concern)

- **`interactive_widget`** — one proposal-card group OR one clarify-widget (form/picker). Carries a
  `widgetState`: `pending | submitted | superseded | stale`, plus (once submitted) the chosen/typed
  value. Ends the turn (D-04: one pending interactive widget per turn).
- **`interaction_result`** — the compact structured "user response" entry (D-16) that appears
  immediately after a successful submit, reads linearly like a normal reply.

### Proposal card group layout (D-05, D-06)

**Design decision (autonomous):** cards render as a **vertical stack** (`flex flex-col gap-2`,
`sm` token), never a responsive row/grid. Rationale: the SAME spec must render identically in the
768px chat column (`22-UI-SPEC.md`) AND inside a 320px-min-width canvas panel (`23-UI-SPEC.md`,
D-08 "never duplicated" — one spec, one layout). A horizontal/grid layout that looks fine at 768px
would collapse illegibly at 320px; a vertical stack is the one layout that is correct at both
widths without introducing a responsive-breakpoint rule this contract would otherwise have to
invent. Each card = the existing catalog `card` node (title + description via its own `propsSchema`)
with ONE `button` node in its `footer` slot — the click target, no other new catalog fields.

```
┌───────────────────────────────────┐
│ Card 1                            │
│ Title                             │
│ Description                       │
│ [ Choose this option ]  ← button  │
├───────────────────────────────────┤
│ Card 2 …                          │
└───────────────────────────────────┘
```

No group-level heading is rendered by the widget itself — the agent's own preceding `text` part
(D-18's interleaved-parts precedent: intro text, then the widget) is the natural question/prompt.
The group container still needs a non-visual label for assistive tech — see Accessibility.

### Clarify-widget layout (D-09)

Unmodified Phase-19 `FormComponent` (`packages/genui/src/catalog/form-component.tsx`) — its own
`grid gap-4` spacing, its own field controls. This phase does not touch that file's layout. The
ONLY new layout surface is the **submitted-state compact view** (see Interaction Contracts), which
REPLACES the live form controls with the existing `key-value-list` catalog primitive (already in
the catalog, zero new components) rendering the submitted field values as a compact `<dl>`.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Proposal card CTA (default, per-card) | "Choose this option" — agent-authored specs SHOULD supply a more specific per-option verb+noun (e.g. "Ship next week") via the same `label`/`aria-label` fields the `button` catalog node already accepts; this is the fallback, not a hardcoded-only string |
| Proposal card CTA — accessible name | `aria-label="{visible label} — {card title}"` (e.g. `"Choose this option — Ship next week"`) — MANDATORY even when the visible label is specific, so repeated/similar labels across sibling cards are never ambiguous out of context for screen-reader users |
| Proposal card group container | `role="group" aria-label="Choose an option"` (static fallback — the agent's own preceding text part is the visible question, per D-18) |
| Clarify-widget submit label | Reuses `FormComponent`'s existing `submitLabel` prop — MANDATORY: the new interactive-widget tool's spec schema MUST declare `submitLabel` as a required, non-empty, context-specific string (e.g. "Send response", "Confirm details"), enforced in code by the tool's Zod/JSON-schema validation at emission time (not prompt guidance), so the bare "Submit" default is never reachable in practice. Mirrors the MANDATORY posture of the proposal-card `aria-label` rule above |
| "Selected" badge (D-06, chosen card) | `Badge variant="default"` + `Check` icon (lucide) + "Selected" |
| "Superseded" badge (D-02, typing supersedes) | `Badge variant="secondary"` + `Ban` icon (lucide) + "Superseded" — inline caption beneath the widget: "You replied by typing instead." |
| "Stale" badge (D-12, newer turn/regenerate) | `Badge variant="secondary"` + `Clock` icon (lucide) + "Stale" — inline caption: "This is no longer the active response." (same string used proactively AND if a stale submit attempt round-trips a 409 — one consistent message, not two) |
| "Submitted" badge (clarify-widget only — see Interaction Contracts for why proposal cards don't need a separate one) | `Badge variant="secondary"` + `CheckCircle2` icon (lucide) + "Submitted" |
| Submitting / in-flight (D-11, cosmetic client lock) | `Loader2` (`animate-spin`, `motion-safe:`) + "Submitting…" — replaces the clicked button/disables the form in place, same visual register as the chat surface's existing "Generating…" row |
| Double-submit conflict (server 409) | "This was already answered." — no action offered (the interaction genuinely cannot repeat) |
| Stale-submit conflict (server 409) | Reuses the Stale badge's own caption: "This is no longer the active response." |
| Validation-error / server rejection (D-10) | "This response couldn't be saved. Please try again." (generic — no schema internals leaked client-side, matches project guardrail: detailed errors logged server-side only) — the widget RE-ENABLES for another attempt (the only inline-error case that allows retry, since the server never persisted a `submitted` state) |
| Compact `interaction_result` — proposal cards (D-16) | `Selected "{card title}"` |
| Compact `interaction_result` — clarify-widget (D-16) | Rendered via the existing `key-value-list` catalog primitive (`aria-label={formTitle ?? "Your response"}`), one row per submitted field, `{field label}: {value}` |
| Clarify-widget submitted compact view heading | "Your response" (`text-xs text-muted-foreground`, sits above the `key-value-list` read-out, paired with the "Submitted" badge) |

No generic "Submit"/"Cancel"/"OK"/"Click here" strings introduced — every new string above is
context-specific, matching the codebase's established copy discipline.

## Interaction Contracts

**Proposal card selection flow (D-05, D-06):** `pending` → user clicks a card's footer button →
immediately (client-side, before any server round-trip) ALL cards in the group disable
(`pointer-events-none`, cosmetic per D-11) and the clicked card shows the submitting row
("Submitting…") in place of its button → on success: the clicked card gets the ring+wash+"Selected"
badge, every OTHER card in the group dims (`opacity-50`, `aria-disabled`), the widget's
`interactive_widget` part is now `submitted` → a `CompactInteractionEntry` (`Selected "{title}"`)
appends immediately after → the existing Phase-22 streaming state machine takes over for the
continuation turn exactly as it does for any new turn (no new "continuation loading" UI — reuse
"Generating…"/`Loader2`/Stop button verbatim).

**Clarify-widget submission flow (D-09, D-10):** `pending` → user fills the unmodified
`FormComponent`, submits → client-side field validation (Phase-19 `validateForm`, unchanged) → on
pass: ALL form controls disable in place (cosmetic, D-11), submitting row replaces the submit
button → on success: the ENTIRE live form is replaced by the compact submitted view ("Your
response" label + `Submitted` badge + `key-value-list` read-out of the submitted values, per D-16
"compact") → a `CompactInteractionEntry` appends after (same `key-value-list` content, in the
transcript) → continuation turn streams exactly as above.

**Typing supersedes a pending widget (D-02):** the composer is NEVER blocked by a pending widget
(D-02, restated). The instant the user's typed message is sent, the pending `interactive_widget`
part transitions to `superseded` client-side (optimistic — no round-trip needed to know a NEW user
message was just sent) — whole group dims (`opacity-50`), the "Superseded" badge + caption appear,
every button/control becomes `aria-disabled` + non-focusable. No "None of these" card is ever
auto-added (D-07) — this transition IS the escape hatch.

**Staleness detection (D-12):** computed reactively client-side from the SAME turn/sibling-version
state Phase-22's D-16 regenerate feature already tracks — a pending widget whose owning turn is no
longer on the ACTIVE sibling path (a regenerate happened, or a strictly newer turn exists) shows the
"Stale" badge proactively, before any submit is even attempted. If a submit is attempted anyway
(race condition — e.g. two tabs), the server's 409 is the authoritative source of truth (D-12) and
the client reconciles to the same stale visual state + caption (defense-in-depth, matches D-11's own
"client-side disable is cosmetic only" posture).

**Double-submit protection (D-11):** the DB-level uniqueness guarantee is the REAL lock; every
client-side disable described above is cosmetic UX only. A 409 from a genuine double-submit (e.g. a
second tab, or a resend after the first request already landed) shows "This was already answered."
with no retry action — the widget reconciles to its true `submitted` state (fetched/pushed from the
server), never left stuck mid-"Submitting…".

**Validation-error / server rejection (D-10):** unlike the double-submit/stale 409s (which mean
"this truly cannot be answered"), a 422-class validation rejection means the attempt itself was
malformed against the stored declared schema — this is the ONE case where the widget RE-ENABLES
(cards un-dim, form controls re-enable) for another attempt, with the inline unboxed error row
appearing above the content (see Design Decisions for why it has no border of its own).

**Canvas parity (D-08):** `InteractiveWidgetBoundary` renders IDENTICALLY inside `GenuiPanelNode`
and `MessageTurn` — same component, same props, differing only in the `variant` passed to the
underlying `GenuiPartBoundary` (bare in canvas, default in transcript, per Component Inventory). A
click in either surface updates the SAME message-part state; both surfaces re-render from that one
source of truth (mirrors Phase 23's existing shared-conversation-state contract, extended to widget
interaction state).

**Keyboard operability:** every card CTA and every form control is a native `button`/`input`
element (no div-with-onClick, project-wide rule) — reachable via Tab, activated via Enter/Space, no
new keyboard model needed beyond what `FormComponent` and the catalog `button` already provide.

## Accessibility

- **Ambiguous-label risk (real, specific to this phase):** repeated per-card CTA labels ("Choose
  this option" × N) are indistinguishable out of context for screen-reader users navigating by
  button list. MANDATORY: `aria-label="{visible label} — {card title}"` on every proposal-card
  button (see Copywriting Contract) closes this gap.
- **Group semantics:** the proposal-card group container is `role="group"` with an `aria-label`
  (falls back to `"Choose an option"` when the agent's own spec doesn't supply a more specific one)
  — so assistive tech announces a related set of choices, not N unrelated buttons.
- **Disabled state is announced, not just visual:** dimmed/inactive cards and disabled form
  controls get `aria-disabled="true"` (removed from tab order) AND `aria-describedby` pointing at
  the widget's status badge text (Superseded/Stale/Submitted) — a screen-reader user hitting a dead
  control hears WHY it's inactive, not silence.
- **Live region:** widget-state transitions announce via the existing restrained
  `aria-live="polite"` region (Phase 22's chat region / Phase 23's canvas region) — "Response
  submitted", "This response is no longer active" — never per-keystroke form-field chatter.
- **Focus management:** a successful submit does NOT move focus (matches `22-UI-SPEC.md`'s existing
  "submitting a message keeps focus in the composer" precedent) — the live-region announcement is
  the signal, not a focus yank.
- **Reduced motion:** the submitting `Loader2` spin and any dim/fade transition are `motion-safe:`
  gated — under `prefers-reduced-motion`, these render in their end state instantly.
- **Color contrast / non-color differentiation:** Superseded vs. Stale vs. Submitted are ALL neutral
  `Badge variant="secondary"` — distinguished from each other by icon (`Ban`/`Clock`/`CheckCircle2`)
  and text, never by color alone. Only "Selected" uses accent color, and it also carries a `Check`
  icon + text, not color alone.
- **Touch targets:** no new touch-target exception needed — card CTA buttons are full-width,
  text-labelled (not icon-only), already comfortably above the 44×44px carry-over threshold via
  their own padding; the existing `22-UI-SPEC.md` icon-only 44px rule is unchanged and inherited
  where it already applies (e.g. the composer Send button, untouched by this phase).

## Design Decisions (Autonomous)

Made without user consultation per the full-auto directive — conservative, consistent with prior
phases, documented here per the researcher's mandate:

1. **Vertical stack, never a responsive grid, for proposal cards** — the same spec must render
   correctly at both the 768px chat column and the 320px-min canvas panel (D-08); one layout that
   works at the narrowest surface beats inventing a breakpoint rule.
2. **`GenuiPartBoundary`'s `bare` variant is bundled into THIS phase**, not deferred — this phase is
   already adding new chrome to the exact component `23-UI-REVIEW.md` flagged as triple-nested;
   shipping more chrome on top of the bug rather than fixing it first would make the regression
   worse and contradicts this phase's own no-more-nesting scope hint.
3. **Validation-error row is unboxed** (icon + text only, no border/background of its own) — it
   lives INSIDE a widget that's already inside one bordering layer (`GenuiCard` or the bare node
   shell); giving the error its own bordered card would recreate a 2-layer nest for exactly the
   class of defect this phase is supposed to avoid. This deliberately differs from the standalone
   `InlineErrorCard` (Phase 22), which replaces an entire turn with no other wrapper and therefore
   needs its own box.
4. **`pending` state shows no badge** — badges communicate a DEVIATION from the default only,
   mirroring `TurnStatusBadge`'s existing "every other status renders nothing" philosophy. A widget
   awaiting interaction is the unmarked common case.
5. **Proposal cards get no separate "Submitted" badge** — the "Selected" badge on the chosen card
   IS the submitted signal for that widget kind (D-06 is explicit: highlight chosen, dim others).
   "Submitted" as its own badge only applies to clarify-widgets, which have no per-option "chosen"
   card to mark.
6. **Clarify-widget's submitted view reuses `key-value-list`** rather than re-rendering the same
   form with every control disabled — compact per D-16, zero new catalog components per D-09, and
   avoids a form-shaped wall of disabled inputs.
