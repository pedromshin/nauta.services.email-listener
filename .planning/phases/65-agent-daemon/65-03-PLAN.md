<objective>
Serve the frozen protocol: a WS server bound to 127.0.0.1 with `x-daemon-token` upgrade auth, a
router that zod-validates both directions, the WS implementation of the permission ask loop, the
single-folder watcher broadcasting `fs.watch.event`, and the `session.*` seam Lane E replaces.

Purpose: this is the surface Lane E dials tonight and the only door into a process that can read,
write and execute on the user's PC. Auth at the UPGRADE (not after) means an unauthorized peer
never gets a socket to send frames on; binding the literal loopback means the door does not exist
on the network at all.

Output: `startDaemon()` usable in-process (ephemeral port for tests/smoke) and via `tsx src/index.ts`
on this Windows PC.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| network → WS upgrade | the ONLY entry point to a process with fs/terminal/git power |
| any local process → 127.0.0.1:8787 | localhost is not a trust boundary by itself: any user process can dial it, so the token is the real gate |
| daemon → clients | outbound frames become Lane E's inputs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-65-11 | Spoofing | auth.ts / upgrade | mitigate | `x-daemon-token` checked at the HTTP upgrade with `crypto.timingSafeEqual`; failure ⇒ 401 + `socket.destroy()`, no WebSocket created. Token read from env only (`readDaemonToken`), min 16 chars, absent ⇒ refuse to boot. Tests: missing header, empty, wrong, near-miss (last char differs), correct-but-with-whitespace ⇒ all rejected. |
| T-65-12 | Elevation of privilege | daemon.ts bind | mitigate | `const HOST = "127.0.0.1"` — a literal, not a config key (R-07: config has no `host` field, so a hostile config file cannot expose the daemon). Test asserts `server.address().address === "127.0.0.1"` and that a connection to a non-loopback local IPv4 is REFUSED. Tunneling is explicitly banned tonight (LANE-CONTRACTS.md). |
| T-65-13 | Tampering | router.dispatch | mitigate | Every inbound frame goes through `parseClientFrame` (65-01) before any handler sees it; every outbound payload through `daemonToClient[type]` inside `Client.send`. No handler ever receives an unparsed payload, and no `ws.send` exists outside `Client.send` — gated by grep in verify. |
| T-65-14 | Spoofing | perm loop | mitigate | `perm.request` is daemon→client only (65-01's direction map), so a client cannot forge a prompt. A `perm.decision` for an unknown/already-resolved requestId is ignored (first decision wins, R-03) — a client cannot replay an old approval into a new ask. |
| T-65-15 | Denial of service | pending asks / frames | mitigate | Pending asks are bounded (max 32 outstanding; beyond that `decide` denies immediately). `ws` `maxPayload` set to 1 MiB. `registry.size === 0` ⇒ ask resolves null instantly (no timeout hold). |
| T-65-16 | Information disclosure | logs | mitigate | Boot log names host/port/root-count only. The token is never logged, never echoed in an error, never in an audit line. Watch events carry paths (already inside a configured root, by construction) — never contents. |
| T-65-17 | Denial of service | watcher | mitigate | ONE root (config-validated inside roots, 65-02). `awaitWriteFinish` collapses Windows partial-write storms. Watcher errors go to `onError` + log, never crash the process (an unhandled chokidar error would take the whole daemon down). |
| T-65-SC | Tampering | npm installs | mitigate | No installs. `ws@8.21.1` and `chokidar@3.6.0` are lockfile-pinned + hoisted (verified). `@types/ws` deferred to the orchestrator with an explicit shim-deletion instruction (R-10). |
</threat_model>
