<objective>
Close the major importer-resolution gap (UAT Gap "Forwarding-sender maps to an importer"). The shipped ingestion path hardcodes IngestionConfig.default_importer_id (00000000-0000-0000-0000-000000000001) instead of resolving the forwarding sender to an importer (D-05). Add an ImporterResolver port + a Supabase find-or-create repository (keyed by sender domain → importer slug), replace the fixed default in the live IngestInboundEmailUseCase with sender-based resolution, and add POST /v1/emails/{id}/reprocess that re-runs ingestion supersede-safely (D-16: prior active extraction records are superseded, never overwritten).

Purpose: Restore true multi-tenant keying (D-05) — every email/attachment/component row is keyed by the sender's resolved importer — and give operators a safe reprocess trigger for an already-ingested email.
Output: ImporterResolver port, SupabaseImporterRepository, sender→importer wiring in the live ingest path, ReprocessEmailUseCase (supersede-then-reingest), POST reprocess endpoint, container wiring, tests.
</objective>

importers table (packages/db/src/schema/importers.ts): columns id (uuid, default random), slug (text, NOT NULL, UNIQUE), name (text, NOT NULL), config (jsonb default {}), created_at. The UNIQUE slug is the find-or-create key. The seeded default importer (migration 0005) has id 00000000-0000-0000-0000-000000000001.
Supabase repo pattern (app/infrastructure/supabase/entity_type_repository.py): `__init__(self, client: Client)`; uses self._client.table("...").select(...).eq(...).execute(); row dicts → frozen domain entities via a _from_row helper. Mirror this style + the existing serializer patterns in email_repository.py.
Supabase upsert idempotency: use `self._client.table("importers").upsert({...}, on_conflict="slug").execute()` then re-select by slug to read back the id, OR select-by-slug-then-insert-if-missing. Must be idempotent under concurrent SNS redelivery (upsert on the UNIQUE slug is the safe path).
IngestInboundEmailUseCase (app/application/use_cases/ingest_inbound_email.py): currently `importer_id = self._config.default_importer_id`. Constructor takes config: IngestionConfig (frozen, default_importer_id). After 04-11 it also injects components, parser_registry, propose_regions. THIS plan adds importer_resolver and replaces the importer_id line with `importer_id = await self._importer_resolver.resolve(parsed.sender_address)`. parsed.sender_address comes from parse_mime.
Reprocess re-trigger key (RESOLVED — do not deviate): IngestInboundEmailUseCase.execute(ses_message_id) takes the BARE SES message id; internally raw_store.fetch(ses_message_id) builds the S3 key via key_for() which PREPENDS the env prefix (S3RawEmailStore.key_for returns f"{prefix}{message_id}"). Email.raw_storage_key stores the FULL key (prefix + ses id), so passing raw_storage_key directly to execute() would double-prefix and 404. Email.message_id is NOT reliable either: ingest sets it to `parsed.message_id or ses_message_id`, i.e. the RFC 5322 Message-ID header when present, which is not the SES id. Therefore: derive the bare SES id as the last path segment of email.raw_storage_key — `ses_id = email.raw_storage_key.rsplit("/", 1)[-1]` (the SES prefix always ends with "/" per settings.ses_s3_prefix, e.g. "inbound/prod/").
ExtractionRepository (app/domain/ports/extraction_repository.py): save, find_by_component_id, supersede_active(component_id) -> None (D-16, already defined). Reprocess supersedes per-component before re-ingest.
ComponentRepository.find_by_email_id(email_id) -> list[Component] (to enumerate components whose extractions must be superseded).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SES From header → importer resolution | The forwarding sender address is attacker-influenced input that selects/creates a tenant row |
| client → POST /v1/emails/{id}/reprocess | Authenticated (X-API-Key) but email_id is attacker-supplied; reprocess mutates extraction state |
| reprocess → confirmed extractions | Reprocess must not destroy human-confirmed values |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-31 | Spoofing | forged From header creates/claims an arbitrary importer | accept (this phase) | No auth on inbound SES yet (Phase 3 boundary); importer is derived deterministically from the sender domain and find-or-create is idempotent. Cross-tenant data exposure is gated by the importer_id key + the read API's tenant guard. Documented limitation until inbound auth exists. |
| T-04-32 | Tampering | reprocess silently overwrites a human-confirmed value | mitigate | ReprocessEmailUseCase calls ExtractionRepository.supersede_active per component (status→superseded), never delete/overwrite of confirmed field data (D-16); the endpoint test asserts the call. |
| T-04-33 | Information Disclosure | reprocess of another tenant's email | mitigate | The endpoint calls _get_tenant_email first (404 on cross-tenant / unknown id) before any supersede or re-ingest. |
| T-04-34 | Denial of Service | malformed/garbage sender domains create unbounded importer rows | mitigate | slug_for_sender normalizes + validates the domain; malformed senders fall back to the default importer rather than creating junk rows; upsert on the UNIQUE slug prevents duplicates. |
</threat_model>
