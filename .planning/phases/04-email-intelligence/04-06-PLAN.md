<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SNS/SES → app | Unauthenticated SNS POST (no X-API-Key by design); payload and S3-referenced bytes are untrusted |
| client → /v1/emails API | Authenticated (X-API-Key) but importer_id is attacker-supplied input |
| S3 → app | Raw email bytes are untrusted (MIME bombs, malformed parts) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-19 | Spoofing | forged SNS Notification triggers ingestion | accept (this phase) | SNS branch already returns 200 on all paths; SNS signature verification is an existing Phase 3 concern, not re-opened here. Ingestion only reads from the SES-controlled S3 key. |
| T-04-20 | Information Disclosure | client passes another tenant's importer_id to GET /v1/emails | mitigate | Read use-cases filter every query by the importer_id argument and the repositories enforce `.eq("importer_id", ...)`; until auth exists (deferred), importer_id is the tenant key — documented limitation. Pydantic validates importer_id as a UUID at the boundary. |
| T-04-21 | Denial of Service | giant MIME / many attachments exhausts memory | mitigate | Cap attachment count and per-attachment size before parsing; per-attachment try/except isolates failures; oversize → mark unsupported. |
| T-04-22 | Tampering | reprocess silently overwrites a human-confirmed value | mitigate | Reprocess calls ExtractionRepository.supersede_active (status→superseded), never delete/overwrite of confirmed field data (D-16); test asserts. |
| T-04-23 | Tampering | malformed attachment crashes the worker | mitigate | Parser (04-04) degrades corrupt files to parse-error Components; decompose marks unsupported types and continues. |
</threat_model>
