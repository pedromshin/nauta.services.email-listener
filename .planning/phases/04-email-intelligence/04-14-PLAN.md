<objective>
Ground proposed region polygons in REAL document coordinates instead of LLM-invented boxes. Today `SegmenterProtocol.segment(page_text, page_index)` hands the model only flat text and asks it to draw a `polygon`; the model has no coordinate information, so `ProposedRegion.polygon` is hallucinated and Phase 5 overlays would render boxes unrelated to where entities sit. Now that 04-13 persists per-word/line token geometry in `content_raw`, change the seam to feed the model coordinate-bearing tokens and have it SELECT which tokens compose each region; the use case then computes each region's polygon as the union of the selected tokens' real bboxes.

Changes:
- `SegmenterProtocol.segment` accepts `tokens: tuple[PageToken, ...]` (each PageToken = index + text + normalized bbox) instead of `page_text`. `ProposedRegion` drops the model-drawn `polygon` and carries `token_indices` (the tokens the model assigned to the region).
- `AnthropicSegmenter` presents the tokens as a numbered list inside `<document_content>` (D-14 preserved) and its `segment_document` tool schema replaces `polygon` with `token_indices` (array of integers).
- `ProposeRegionsUseCase` reads each page's tokens from 04-13 `content_raw`, passes them to `segment()`, and computes each child region's polygon as the union of its selected tokens' bboxes (4-corner normalized polygon), falling back to the page polygon when no valid tokens are selected.

Purpose: region overlays in Phase 5 and any downstream geometry are anchored to actual token coordinates. Preserve every 04-05 robustness guarantee.
Output: token-based segmenter seam, updated tool schema + prompt, token-grounded polygon computation in the use case, and rewritten tests proving region polygons equal the union of member tokens' bboxes (multi/overlapping/nested), junk → [], and D-14/retry/cost-guard intact.
</objective>
