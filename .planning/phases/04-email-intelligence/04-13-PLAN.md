<objective>
Stop discarding the granular coordinate geometry the PDF pipeline already has. Today `_component_from_ocr` collapses Textract per-word bboxes into one page-spanning rectangle via `_bbox_union` and sets `content_raw=None`; `_component_from_text` hardcodes `_FULL_PAGE_POLYGON` and drops the pdfminer `LTTextContainer.bbox` entirely. As a result the only persisted geometry is a full-page box, so Phase 5 region overlays would render boxes with no relation to where text actually sits, and 04-14 cannot ground region polygons in real coordinates.

This plan retains word/line bbox geometry on every page Component:
- OCR path: persist the per-word OcrWord bbox array (already normalized 0-1, top-left origin) into `content_raw`.
- Text-layer path: capture each `LTTextContainer.bbox`, normalize to 0-1 and flip the Y axis to a top-left origin (matching the Textract/D-12 convention), persist the per-line/block tokens into `content_raw`, and replace the hardcoded full-page polygon with the union of the real element bboxes.

Purpose: give Phase 5 overlays and 04-14 region grounding real coordinates to draw against. Non-breaking — only ADDS `content_raw` (already a jsonb column) and refines the page polygon; no migration, no schema change, no API change.
Output: a `pdf_parser.py` that persists token geometry on both paths, plus tests proving token bboxes are persisted for text + OCR pages, are normalized 0-1, and that garbage pages still degrade to a full-page polygon without raising.
</objective>
