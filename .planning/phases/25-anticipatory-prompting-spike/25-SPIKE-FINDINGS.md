# Phase 25 Spike Findings: Anticipatory Prompting (ANTIC-01/ANTIC-02)

**Captured:** 2026-07-06
**Spike scope:** ANTIC-01 (trigger/heuristic layer over chat+canvas state) + ANTIC-02 (appropriateness eval + frequency cap, explicit-accept only)
**Exit criterion (D-03):** a documented go/no-go recommendation, not a shipped guarantee.

## Verdict

ship-with-conditions

The trigger → independent-gate-chain → explicit-accept mechanism behaves exactly as designed against all three fixtures: it is structurally false-positive-averse (both gates default to suppression, a judge error suppresses rather than fires, the flag defaults OFF), it never bypasses explicit user consent, and its two gates are provably independent rather than redundant. What the spike does NOT establish is whether the mechanism is worth shipping on live traffic — that requires a real appropriateness judge, durable state, and live UI wiring, none of which this spike built. Ship the *mechanism* forward into a follow-up phase, gated on the four conditions in "Seams a shippable feature needs" (items 1-4) being addressed before the flag is ever flipped on for a real user.

## What was built

A real, working, flag-gated pipeline — not a paper study (D-01). Everything below is dark by default: `ANTICIPATORY_PROMPTING_ENABLED: bool = False` in `apps/email-listener/app/settings.py` is the single global off switch (D-12), and `EvaluateAnticipatoryCandidates.evaluate()` returns immediately with zero candidates and zero lifecycle events when it is `False` — the trigger layer is never even invoked.

- **Triggers (D-04, ANTIC-01)** — `apps/email-listener/app/domain/anticipatory/triggers.py`. Three deterministic, no-ML functions sharing one `AnticipatoryTrigger` Protocol signature, run by `run_triggers()`: `detect_idle_after_genui` (user inactive ≥45s after a settled genui/interactive-widget turn), `detect_completed_artifact` (a settled canvas panel carrying a `next_best_action`), `detect_ambiguous_intent` (last user message is ≤4 tokens or matches a frozen vague-phrase set). Each is read-only over `AnticipatoryStateSnapshot` (D-06) — every collection field is a tuple, enforced at the type level, not by convention.
- **Typed candidate (D-05)** — `apps/email-listener/app/domain/anticipatory/candidate.py`. A trigger never emits free text; it emits an `AnticipatoryCandidate` (trigger id, proposed prompt text, rationale, source refs) plus the `AnticipatoryLifecycleEvent`/`AnticipatoryStateSnapshot`/`SourceStateRef` contracts every downstream stage shares.
- **Appropriateness eval (D-07/D-09, gate #1)** — `apps/email-listener/app/infrastructure/llm/anticipatory_judge_adapter.py` (`BedrockAppropriatenessJudgeAdapter`, Haiku, forced-tool-use, `max_tokens=256`) scores a candidate 0-1 against a neutral context summary (never raw chat text, T-25-03) and — critically — resolves ANY error, timeout, or invalid output to `score=0.0, reason="judge_error_suppress"`, the opposite failure direction from the code-island judge's "safe default = first candidate." `apps/email-listener/app/domain/anticipatory/stubs.py` (`StubAppropriatenessJudge`) is the deterministic double this spike's tests — and this findings doc's evidence — run against; no live Bedrock call was made anywhere in this spike's test suite (D-09).
- **Frequency cap (D-10/D-14, gate #2)** — `apps/email-listener/app/infrastructure/anticipatory/in_memory_cap_store.py` (`InMemoryAnticipatoryCapStore`), an in-process `conversation_id -> shown timestamps` dict enforcing both a per-window and a per-day ceiling. D-14 (planner decision): no new DB table — the port (`app.domain.ports.anticipatory_ports.AnticipatoryCapStore`) is the real architectural boundary; a production adapter would project the existing `chat_run_events` substrate instead of living in-process.
- **Gate chain (D-08) — `apps/email-listener/app/application/use_cases/evaluate_anticipatory_candidates.py`** (`EvaluateAnticipatoryCandidates`). The cap is checked *before* the judge purely as a cost optimization (a capped candidate never bills a Bedrock call) — never a substitution; a cap denial is never overridden by what the eval would have said, and vice versa. Proven independent by a dedicated test showing the SAME candidate resolves to all three outcomes purely as a function of which single input changes.
- **Explicit-accept reuse (D-11)** — `to_proposal_card_declaration()` in the same module maps a surviving candidate onto the UNCHANGED Phase-24 proposal-card declaration shape (`derive_declared_response_schema("proposal_cards", ...)`, `run_chat_turn_widgets.py` untouched). Nothing fires without a real user click through the existing explicit-accept submit round trip.
- **Lifecycle log (D-13)** — every candidate's `proposed -> {suppressed_by_eval | suppressed_by_cap | shown} -> {accepted | dismissed}` transition is recorded as an `AnticipatoryLifecycleEvent` and emitted via `structlog` (`anticipatory_lifecycle`). `record_candidate_outcome(candidate, "dismissed", cooldowns=...)` registers a per-trigger cooldown that suppresses that trigger's candidates on a later `evaluate()` call passed the same cooldown set.
- **DI wiring** — `apps/email-listener/app/container.py` registers the judge adapter, the in-memory cap store, and `EvaluateAnticipatoryCandidates` as real, resolvable providers; `create_app()` boots cleanly with the whole pipeline wired and the flag confirmed `False`.

## Evidence (fixture matrix)

Captured 2026-07-06 from `apps/email-listener/app/application/use_cases/__tests__/test_anticipatory_spike_harness.py::test_build_spike_outcome_matrix_matches_expected_and_prints_evidence` (`build_spike_outcome_matrix()`), which drives the REAL `EvaluateAnticipatoryCandidates` gate chain over each of the three Plan 25-01 fixtures across four scenarios. All 16 harness tests pass deterministically (`uv run pytest ... -s`, `--no-cov`); no live Bedrock call anywhere.

| Fixture | A: appropriate + cap room (score=0.9, empty cap) | B: appropriate + capped (score=0.9, cap at window limit) | C: inappropriate + cap room (score=0.3, empty cap) | D: flag OFF (`enabled=False`) |
|---|---|---|---|---|
| `idle_after_genui` | `shown` | `suppressed_by_cap` | `suppressed_by_eval` | `none` (zero candidates/events) |
| `completed_artifact` | `shown` | `suppressed_by_cap` | `suppressed_by_eval` | `none` (zero candidates/events) |
| `ambiguous_intent` | `shown` | `suppressed_by_cap` | `suppressed_by_eval` | `none` (zero candidates/events) |

What the matrix shows:

- **Every gate outcome is reachable and each fixture fires exactly one trigger** — no cross-trigger contamination (a hand-built "quiet" snapshot with no idle/panel/ambiguity signal also yields zero candidates, per Plan 25-01's own tests).
- **False-positive suppression behavior is structural, not incidental.** Column B proves the cap independently vetoes a candidate the judge would have approved (score 0.9); column C proves the judge independently vetoes a candidate the cap would have allowed. Neither gate can be bypassed by the other passing — proven per-fixture here and originally proven in Plan 25-02's dedicated "independence" test.
- **Cap independence and cost posture:** the cap check runs first and is free (no Bedrock call); a cap-denied candidate never reaches the paid judge call. Judge calls that do happen use Haiku with `max_tokens=256` and a 30s timeout (`ANTICIPATORY_JUDGE_TIMEOUT_SECONDS`), and any judge failure resolves to suppression, not to a shown candidate.
- **Flag-OFF is absolute:** column D produces literally zero lifecycle events for every fixture — confirmed with judge/cap-store doubles that raise `AssertionError` if ever invoked, so "OFF means OFF" is enforced by the test, not just asserted by inspection.
- **Intrusiveness posture:** a `shown` candidate (column A) only ever becomes a Phase-24 proposal-card declaration — no side effect fires until a real user click resolves through the unchanged `SubmitWidgetInteraction` round trip; a `dismissed` outcome registers a cooldown that silences that trigger on the next evaluation of the same conversation state.

## Seams a shippable feature needs

1. **Live-Bedrock appropriateness scoring + a real false-positive-rate study on live traffic** (deferred per the EVAL-LIFT/ISO-RUN convention; CONTEXT Deferred Idea "Live-Bedrock appropriateness-eval measurement + a real false-positive-rate study on live traffic"). This spike proves the judge *contract* (fail-toward-suppress, cheap, timeout-bounded) but every score in the evidence table above came from `StubAppropriatenessJudge`, never `BedrockAppropriatenessJudgeAdapter` against a real model. The 0.75 threshold is untested against real judge score distributions.
2. **Durable cap + lifecycle persistence** (D-10's "survives reload" requirement; D-14's explicit deferral). `InMemoryAnticipatoryCapStore` is process-local and does not survive a restart or span multiple app instances. A shippable feature needs the cap state (and the D-13 lifecycle log) projected off the real `chat_run_events` substrate, not an in-process dict — this is the single largest architectural gap between "spike" and "feature."
3. **The live observation adapter** assembling a real `AnticipatoryStateSnapshot` from actual Phase-22 run events + Phase-23 canvas store + Phase-24 widget-interaction state (CONTEXT code-context "Integration Points" — the chat loop as the observation + candidate-production site). This spike's snapshots are hand-scripted fixtures (D-02, by design); nothing in this spike reads live conversation state.
4. **Live web wiring of a shown candidate into the existing Phase-24 proposal-card emit path** (`apps/web/src/app/chat` — CONTEXT "Integration Points"). `to_proposal_card_declaration()` is proven to round-trip through `derive_declared_response_schema`, but no code in this spike calls it from the live chat turn loop or renders it to a browser — the emit-to-UI wire is unbuilt.
5. **Dismissal-cooldown durability across reload.** `record_candidate_outcome`'s cooldown registration is a caller-owned in-memory `set[TriggerId]` (Plan 25-02's key-decisions), proven correct within a single test's lifetime but with no persistence story — a page reload or new process would forget every dismissal, letting a just-dismissed trigger fire again immediately.
6. **Cross-conversation / global anticipatory context** (CONTEXT Deferred Idea, explicitly out of scope for a single-chat spike; north-star, not a near-term gap) — today every gate is scoped to one `conversation_id`; a real feature may eventually want cross-conversation rate limiting or personalization, deliberately not attempted here.
7. **Learned/ML trigger models and personalization** (CONTEXT Deferred Idea) — all three triggers are deterministic and rule-based by design (D-04); replacing or augmenting them with a learned model is explicitly future work, not a gap in this spike's own scope.

## Risks & open questions

- **False-positive risk on live traffic is the primary open question.** The mechanism is provably suppression-biased in fixture space, but fixture space is not live-user space: real appropriateness judge scores, real idle patterns, and real ambiguous-intent phrasing will differ from the three scripted scenarios. Seam 1 must close before this risk is retired.
- **Threshold/cap tuning is a documented placeholder, not a validated value.** `ANTICIPATORY_APPROPRIATENESS_THRESHOLD = 0.75` was chosen (Claude's discretion, per settings.py's inline comment) to be "deliberately conservative/high" because D-07 mandates biasing hard toward not prompting — it was tuned to make the fixtures behave sensibly, not fit to a live score distribution. `ANTICIPATORY_IDLE_THRESHOLD_SECONDS = 45.0` was chosen to be "long enough that a user mid-read isn't interrupted, short enough to still feel anticipatory" — a judgment call, not a measured value. `ANTICIPATORY_CAP_PER_WINDOW = 1` per `ANTICIPATORY_CAP_WINDOW_MINUTES = 10`, plus `ANTICIPATORY_CAP_PER_DAY = 3`, were chosen so the two ceilings are independently reachable in a short fixture-driven test window; a live conversation's cadence may call for looser or tighter numbers. All four values live in `apps/email-listener/app/settings.py` as named, overridable settings, not magic numbers, so they are cheap to retune later.
- **Judge cost at scale is untested.** Haiku at `max_tokens=256` per candidate is cheap per-call by design (D-09, mirroring `GENUI_CODE_JUDGE`'s posture), and the cap-first ordering already skips the judge call for capped candidates, but no cost projection exists for "N triggers firing across M concurrent live conversations" — this is a straightforward calculation to do once seam 3 (live observation) exists and real trigger-firing rates are known, but it was out of scope for a fixture-driven spike.
- **The in-memory cap store's absence of persistence is a known, planner-approved shim (D-14), not an oversight** — flagged again here because it is the seam most likely to be under-estimated: without it, a real deployment restart would silently reset every conversation's frequency cap to zero, undermining the very guarantee ANTIC-02 exists to provide.

## What this spike did NOT prove

- **Live model behavior.** No test in this phase issued a real Bedrock call. `BedrockAppropriatenessJudgeAdapter`'s schema/parsing/fail-toward-suppress behavior is unit-tested (Plan 25-02), but its actual scoring judgment on real proposed prompts was never observed.
- **A real false-positive rate on real traffic.** The evidence table above is a correctness proof of the gate-chain *mechanism* (every wiring path behaves as designed), not a measurement of how often a live user would find a shown prompt unwelcome. Those are different claims, and only the first is substantiated here.
- **Whether the three starter triggers (idle-after-genui, completed-artifact, ambiguous-intent) are the RIGHT triggers** for real usage, versus merely three triggers deterministic enough to spike against. No usage data informed their selection beyond the R3 research framing.
- **Any UI/UX validation** of what a shown proposal card feels like in a live chat — this spike never rendered a candidate to a browser (seam 4).
