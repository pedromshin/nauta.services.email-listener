# Phase 22 — UI Design Contract

> Visual and interaction contract for `/chat` — persistent, streamed conversation with progressive
> genui, table-stakes chat mechanics, multi-provider model picker, and cost circuit breaker.
> Generated by gsd-ui-researcher (autonomous session — no user prompts; all choices derived from
> `22-CONTEXT.md`'s 28 locked decisions + the existing app's established shadcn/frosted design
> system). Verified by gsd-ui-checker.

**Extends, does not replace:** the app shell (`AppSidebar`, frosted rail, next-themes light/dark,
`SidebarInset` content slot) and the Studio surface's state-chrome idioms (`Loader2 animate-spin` +
text, `Badge` for status chips, `role="alert"` destructive-tinted banners, 55/45
`ResizablePanelGroup`). No new brand colors, radii, or fonts are introduced anywhere in this spec.

## Layout & Structure

`/chat` renders inside the existing root `SidebarInset` slot (app shell rail stays outermost, per
`layout.tsx`). Two states:

1. **Home screen** (D-13, no conversation selected): conversation rail (below) + a centered empty
   state in the main column — mirrors the `EmptyState` pattern already used in
   `entities-gallery.tsx` (icon + `text-sm font-semibold` heading + `text-sm text-muted-foreground`
   body, `py-24` vertical rhythm) but larger (this is a primary landing surface, not a sparse-list
   state) — see Copywriting Contract.
2. **Conversation view**: rail + main column split `flex`. Main column = top toolbar (model
   picker + cost meter) → scrollable message list (flex-1) → composer (shrink-0, pinned bottom).

**Conversation rail (D-11)** — own collapsible rail nested inside `/chat`, NOT a second
`SidebarProvider`. Reusing `@nauta/ui/sidebar`'s `SidebarProvider` would collide with the app
shell's shared `sidebar:state` cookie (global key, single value) — two providers cannot coexist
safely. Build the rail from `@nauta/ui/collapsible` (Radix `Collapsible`) + local component state
persisted to `localStorage["chat:rail:collapsed"]`, independent of the app-shell cookie.

| Element | Spec |
|---------|------|
| Rail width (expanded) | `280px` fixed |
| Rail width (collapsed) | `0px` (slide-out, not icon-rail — conversation titles aren't icon-representable) |
| Rail chrome | `border-r border-border/50 bg-background/70 backdrop-blur-md` — exact match to `AppSidebar`'s frosted treatment |
| Rail collapse transition | `transition-[width] duration-200 ease-in-out`; wrap in `motion-safe:` — under `prefers-reduced-motion`, toggle instantly (no animated width) |
| New-chat affordance | `Button` `variant="default"` full-width at rail top, `size="sm"`, `Plus` icon + "New chat" label — most prominent element in the rail (D-13) |
| Row content | icon-free text row: title snippet (truncate), relative timestamp (`text-xs text-muted-foreground`), overflow `DropdownMenu` (rename / delete) revealed via a `MoreHorizontal` trigger that is **always rendered** (not hover-only — see Accessibility) |
| Active row | `bg-primary/10 text-primary` — identical treatment to `AppSidebar`'s active-nav-item styling (D-20 continuity, single accent hue) |

**Message column max-width:** `max-w-3xl mx-auto` inside the scrollable list and the composer wrapper
(768px reading column) — standard chat-product convention, improves the 1.5 line-height body
readability; does not affect the rail or toolbar which stay full-bleed.

## Spacing Scale

Declared values (must be multiples of 4) — the project-standard 8-point scale, unchanged:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge internal padding |
| sm | 8px | Compact spacing: turn action-row icon gaps, interleaved-part vertical gaps (D-18) |
| md | 16px | Default element spacing: message bubble padding, composer padding, rail row padding |
| lg | 24px | Section padding: toolbar height padding, empty-state icon-to-heading gap |
| xl | 32px | Layout gaps: gap between distinct message turns |
| 2xl | 48px | Major section breaks: not used in chat (no phase-22 surface needs this) |
| 3xl | 64px | Page-level spacing: home-screen empty-state vertical padding (`py-16`→`py-24` range) |

Exceptions:
- Composer send/stop button and rail overflow-menu trigger: **44×44px** minimum hit target
  (touch-target exception, overriding the visual 32px icon-button size — use `size-11` hit area
  with a smaller visual icon centered inside, consistent with WCAG 2.5.5).
- Textarea min-height: `44px` (single line), auto-grows to `max-h-52` (208px ≈ 13 lines) then
  scrolls internally — not on the 8-point scale by necessity (line-height driven), documented here
  as the one intentional non-multiple-of-4 exception.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 12px (`text-xs`) | 600 semibold | 1.4 — badges, cost-meter figures, model capability tags, timestamps |
| Body | 14px (`text-sm`) | 400 regular | 1.5 — message text, composer input, rail row titles |
| Heading | 16px (`text-base`) | 600 semibold | 1.3 — conversation title in toolbar, rail section label ("Conversations"), model-picker entry name |
| Display | 24px (`text-2xl`) | 600 semibold | 1.2 — home-screen empty-state heading only |

Exactly 2 weights declared (400/600), matching the codebase-dominant pattern (verified: `font-semibold`
is the majority weight class across `apps/web/src`; `font-medium` appears only in a minority of
pre-existing components and is NOT introduced net-new by this phase). Markdown-rendered assistant
prose (CHAT-07) must map its own heading levels down into this same 2-weight system — e.g. markdown
`##`/`###` render as `text-base font-semibold` (Heading role), never introduce a third weight for
rendered markdown.

## Interaction Contracts

**Streaming state machine (STREAM-01, per assistant turn):**
`idle → streaming → (completed | stopped | failed | cost_capped)`. Composer textarea is disabled
exactly while the *active* turn is `streaming` (CHAT-06); it re-enables immediately on any terminal
state. Visible generating indicator = (a) a blinking text caret (`▍`, CSS `animate-pulse` or a
custom blink keyframe, `motion-safe:` gated) at the tail of the growing text, AND (b) a status row
above the composer: `Loader2 animate-spin` + "Generating…" + the Stop button — this reuses the
exact `Loader2`-plus-text idiom from `generation-state-chrome.tsx` rather than inventing a new
motif.

**Stop (D-15, CHAT-03):** Stop button replaces Send in the same physical slot (no layout shift).
Clicking stop transitions the turn to `stopped`; the partial content already rendered stays in the
thread, tagged with the "Stopped by user" `Badge`. This is a normal turn state, not an error — no
destructive styling.

**Regenerate (D-16, CHAT-04):** Produces a new sibling version of the assistant turn (never
replaces). `SiblingNav` (`‹ N/M ›`) appears in the `TurnActionRow` once `M > 1`; navigating siblings
swaps the rendered turn content only — it does not re-fetch or affect other turns. Only the
currently-active sibling feeds forward context to subsequent turns (server-side concern, but the UI
must make "active" visually unambiguous — the counter is the only indicator needed, no extra
chrome).

**Failed turn (D-19, CHAT-05):** Renders `InlineErrorCard` in place of the assistant turn's content
(same position, same turn), with a `Retry` action. The user's composer draft (whatever they've
typed for their *next* message) is never touched by a failure elsewhere in the thread — draft state
and turn state are fully decoupled.

**Cost circuit breaker (STREAM-03, D-20..D-23):**
- Pre-turn fail-closed block (D-21): renders `CostCapBlockedCard` instead of ever starting the
  turn — same visual family as `InlineErrorCard` but no Retry action (see Copywriting Contract).
- Mid-stream abort at per-turn cap (D-21): identical treatment to Stop, except the marker reads
  "Cost-capped · partial response" instead of "Stopped by user". These two markers are mutually
  exclusive per turn and share one badge slot.
- Session cost meter (D-23): always visible but subtle (`text-xs text-muted-foreground`) in the
  toolbar — never modal, never blocking.

**Interleaved typed parts (D-18, D-02):** A turn's parts (text / `emit_ui_spec` tool-call+result /
text / …) render in emission order inside one `MessageTurn` container — no per-part bubble, all
parts share the turn's outer spacing. Each genui part renders inside a bordered `Card`
(`border border-border rounded-lg p-4 my-2`) to visually separate widget content from surrounding
prose; text parts render as plain flowing markdown with no container.

**Progressive genui rendering (D-17, STREAM-02):** `GenuiPartBoundary` renders whatever subtree of
the partial JSON already validates against the schema live, and renders a generic placeholder for
each not-yet-arrived child: three stacked `Skeleton` bars (`animate-pulse rounded-md bg-muted`,
widths `w-full`, `w-5/6`, `w-2/3`) unless a more specific per-widget-type skeleton is worth the
effort at implementation time (executor's call — the 3-bar generic is the required minimum). If the
final spec fails validation once complete, the existing `SAFE_FALLBACK_SPEC` path applies unchanged
(no new fallback UI to design).

**Model picker (D-04..D-10):** Trigger button in the toolbar shows the current model's short name.
Opens a `Command` (cmdk) list grouped by provider (Bedrock / OpenRouter / Browser). Each row shows
name, capability flags, cost estimate or "Local · Free", and the "Best for" caption (see Copywriting
Contract) — all always visible in the row, not hidden behind a hover or a second click (D-06:
"effortless" choice). Selecting a browser-locus model that hasn't been loaded yet triggers the
WebLLM loading sequence inline in the row (Progress component) before it becomes selectable.
Remembered-last-used model (D-10) is the default for a new conversation and carries a small
"Recommended" outline in `primary` (the one picker element allowed to use the accent color, since
it is the honest "this is what you used last" signal, not decorative).

**Delete / rename (D-12, D-14):** Delete always requires the `AlertDialog` confirm (hard delete, no
soft-delete/undo). Rename is inline-only (click title or an Enter/F2 keyboard shortcut on a focused
row) — commits on blur or Enter, cancels on Escape, no separate save button.

## Accessibility

- **Live regions:** a visually-hidden `aria-live="polite"` status element announces *state
  transitions only* — "Generating response", "Response complete", "Response stopped by user",
  "Response failed", "Cost limit reached" — never the growing delta text itself (announcing every
  streamed token would spam screen readers; this deliberately deviates from wrapping the full
  message in `aria-live`, which the codebase's existing `generation-state-chrome.tsx` idiom does
  for a one-shot state, not a continuously-mutating stream).
- **Errors:** `InlineErrorCard` and `CostCapBlockedCard` use `role="alert"` (overrides the polite
  region), matching the existing fallback-banner convention exactly.
- **Focus management:** submitting a message keeps focus in the composer textarea (never moves it
  away). Creating a new chat (D-13) moves focus into the composer. Stop and Send occupy the same
  tab-stop position so focus doesn't jump when the button morphs.
- **No hover-only affordances:** the rail's rename/delete overflow trigger and the `TurnActionRow`
  (copy/regenerate/sibling-nav) render always-visible (subtly, `text-muted-foreground`), not
  hover-reveal-only — required for keyboard and touch users, a deliberate divergence from common
  hover-chrome chat-UI patterns.
- **Touch targets:** Send/Stop and the rail overflow trigger use a 44×44px hit area (see Spacing
  Scale exception) even though the visual icon is smaller.
- **Reduced motion:** rail collapse/expand transition, the streaming cursor blink, and skeleton
  pulse are all wrapped in `motion-safe:` — under `prefers-reduced-motion`, these render in their
  end state without animation.
- **Keyboard:** Enter sends (Shift+Enter = newline, CHAT-06); rename commits on Enter/cancels on
  Escape; all interactive rail rows and turn actions are native `button`/`Link` elements (no
  div-with-onClick), preserving default Tab order and Enter/Space activation.
- **Color contrast:** no new colors — all text/background pairs inherit the already-shipped
  shadcn token contrast ratios (unchanged from the rest of the app).
